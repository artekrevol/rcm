# Replit QA Audit — End-to-End Verification of Recent Billing Sprint Changes

## Purpose of this audit

Over the past several days, multiple sprints have shipped changes affecting the billing pipeline: rules engine wiring, CCI ingestion, snapshot bug fixes, extraction history hooks, white-screen Radix Select fix, schema additions for Chajinel tenant, agency-billed model, soft delete groundwork, Medicare locality data, plus various smaller fixes. The volume of changes is high enough that I am genuinely concerned things may have been declared "complete" without proper verification at all layers.

This audit's job is to catch the bugs BEFORE the May 12 Rosemary meeting and before the Daniela kickoff. The standard is verifiable proof, not assertion. Every claim of "this works" must be accompanied by SQL output, EDI output, screenshots, HTTP response codes, log snippets, or some other piece of independently-verifiable evidence.

A failure to find anything wrong is also a valid result — but it has to be a result based on actual testing, not "I read the code and it looks right."

## Standing constraints — read these before starting

1. **Production database is the source of truth.** All SQL queries run against `PRODUCTION_DATABASE_URL`, not the dev / Replit Postgres. If you cannot connect to production, stop and flag that.

2. **Code reading is not testing.** "I checked the code and the logic looks correct" is not an acceptable answer for any item in this audit. Every assertion needs a runtime check.

3. **Be honest about uncertainty.** If you cannot verify something for any reason — production access denied, data unavailable, environment not reproducible — say so explicitly. Do NOT fabricate verification results. Do NOT skip items silently. A clearly-flagged "could not verify because X" is far more useful than a fabricated success.

4. **Use the standing Production Verification format** for every section. Each item gets: what was checked, the actual command/query/action used, the actual result observed (pasted verbatim, not summarized), and a pass/fail/inconclusive determination.

5. **Do not modify production data** during this audit unless explicitly identified as part of testing (e.g., creating a test claim is fine; running UPDATE statements that change real records is not).

6. **Take screenshots** for any UI verification. Logged-in views, error states, before/after comparisons. These are part of the deliverable.

7. **Capture full HTTP request/response cycles** for any API verification. Method, URL, headers, body, status code, response body, response time. This is proof endpoints actually behave as claimed.

8. **Do not mark anything "complete" until verification has actually run.** If a section of this audit can't be tested for any reason, mark it INCONCLUSIVE with the reason — do not mark it PASS based on assumption.

#### 1.2 Data integrity checks

```sql
-- All claims have a status
SELECT COUNT(*) FILTER (WHERE status IS NULL) AS null_status,
       COUNT(*) FILTER (WHERE status NOT IN ('draft','submitted','accepted','rejected','paid','denied','exported','archived')) AS invalid_status,
       COUNT(*) AS total
FROM claims;
-- null_status and invalid_status must both be 0

-- All claims have an organization_id (tenant integrity)
SELECT COUNT(*) FILTER (WHERE organization_id IS NULL) AS missing_org_id,
       COUNT(DISTINCT organization_id) AS distinct_orgs
FROM claims;
-- missing_org_id must be 0

-- All claims with last_risk_factors are evaluated
SELECT COUNT(*) FILTER (WHERE last_risk_evaluation_at IS NULL AND last_risk_factors IS NOT NULL) AS factors_without_eval_timestamp,
       COUNT(*) FILTER (WHERE last_risk_evaluation_at IS NOT NULL AND last_risk_factors IS NULL) AS eval_without_factors
FROM claims;
-- Both should be 0 — internal consistency

-- Patients all have organization_id
SELECT COUNT(*) FILTER (WHERE organization_id IS NULL) AS orphan_patients FROM patients;
-- Must be 0

-- Service lines all reference valid claims
SELECT COUNT(*) AS orphan_service_lines
FROM claim_service_lines csl
LEFT JOIN claims c ON c.id = csl.claim_id
WHERE c.id IS NULL;
-- Must be 0

-- CCI data integrity
SELECT ncci_version, COUNT(*) AS rows, MIN(effective_date), MAX(effective_date)
FROM cci_edits
GROUP BY ncci_version
ORDER BY ncci_version;
-- Should show 2026Q1 (~1.82M, max effective_date 2026-01-01) and possibly 2026Q2 if re-ingested with v321r0

-- Tenant isolation check
SELECT 
  organization_id,
  COUNT(*) AS payer_count
FROM payers 
GROUP BY organization_id
ORDER BY organization_id NULLS FIRST;
-- Confirm chajinel-org-001 has its own payers AND no leakage from other orgs
```
