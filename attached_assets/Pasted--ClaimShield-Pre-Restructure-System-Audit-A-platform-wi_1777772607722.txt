# ClaimShield — Pre-Restructure System Audit

A platform-wide architectural restructure is being planned (per `ClaimShield_Architecture_Restructure_Scope.docx`). Before any restructure work begins, we need a comprehensive, accurate, current map of what the system actually is today — schema, data flows, module boundaries, integrations, and intricacies.

This prompt is **read-only**. No code changes. No schema changes. No migrations. No refactors. The deliverable is documentation.

The goal is for the human reviewer to be able to read these docs and have complete ground-truth knowledge of the system before architectural decisions are made.

---

## Hard rules

1. **No mutations of any kind.** No DDL, no UPDATE, no INSERT, no DELETE, no migrations, no refactors, no "small fixes," no dependency upgrades. If you notice a bug, document it — do not fix it.
2. **No new feature work.** The only output is documentation files in `docs/architecture/system-audit/`.
3. **Document what exists, not what should exist.** If something is broken, stubbed, or incomplete, say so plainly. Do not present aspirational state as current state.
4. **Cite line numbers and file paths** for any claim about how the system works. A reviewer must be able to verify any statement by opening the referenced file.
5. **Note "I don't know" where applicable.** If a behavior cannot be confirmed by reading code or running a read-only query, mark it as `UNVERIFIED` or `UNKNOWN`. Do not guess.

## Section 1 — Database schema inventory (`01-database-schema.md`)

Produce a complete schema inventory by querying `information_schema` and `pg_catalog`. Do not rely on memory or ORM definitions — query the live database.

For each table:

- Table name and schema
- Approximate row count (`SELECT COUNT(*) FROM <table>`)
- All columns with: name, data type, character_maximum_length where applicable, is_nullable, column_default
- Primary key columns
- All foreign key relationships (source column → target table.column, on-delete behavior)
- All unique constraints
- All check constraints
- All indexes (name, columns, unique, partial)
- Any triggers attached
- Any RLS policies (likely none yet)

For each view: definition.

For each function/stored procedure: name, signature, body length, brief purpose if discernible.

For each sequence: name, current value, owner column.

**Group tables by apparent functional area** — e.g., "Intake (leads, calls, messaging)", "Billing (claims, patients, providers)", "Reference data (HCPCS, ICD-10, CPT/RVU, CCI edits)", "Auth (users, sessions)", "Multi-tenancy (organizations, …)". Note any tables whose function is unclear from name alone.

**Specifically confirm or refute** the following claims from the architecture doc and the in-flight multi-tenancy work — these are the load-bearing assumptions for upcoming restructure work:

- Does a `payers` table exist? If yes, does it have an `organization_id` column?
- Does `practice_settings` exist? Does it have a `billing_model` column?
- Does `patients` have a `leadId` column?
- Does `claims` have `rendering_provider_id`, `billing_provider_id`, etc., or are providers attached differently?
- Do `org_types` and `org_type_field_specs` tables exist (from the earlier multi-tenancy work)? If yes, what's their content?
- Does `org_voice_personas` exist? What columns?
- What is the data type of `organizations.id`?
- What is the data type of `claims.id`, `patients.id`, `providers.id`, `payers.id` (if exists), `leads.id`?

End the file with a "**deprecation candidates**" subsection listing any tables that look like dead code, prior migration leftovers, or experiments that didn't ship. For each, note: who likely created it, whether it has rows, whether anything in `server/` references it.

## Section 2 — Data flows (`02-data-flows.md`)

For each of the major data flows below, produce:

1. A textual narrative of the flow, step by step
2. The list of files/functions/routes involved with line-number citations
3. A Mermaid diagram showing the entities and transitions
4. Any branching logic (per role, per practice type, per feature flag)
5. Known gaps or stubbed steps

Flows to document:

**Flow A — Lead capture to lead record.**
How does a lead enter the system? Vapi voice call inbound, web form, manual creation, CSV import, other? Trace from entry point to the row landing in `leads` (or whatever the table is called).

**Flow B — Vapi voice call lifecycle.**
Full lifecycle: inbound call → assistant override construction → call connection → end-of-call webhook → transcript capture → field extraction → lead creation/update → status transitions. Include the role of `org_voice_personas` (if it exists) and how the system prompt is built.

**Flow C — Lead to patient conversion.**
How does a lead become a patient? Manual button click, automatic on qualification, both? Trace `syncPatientToLead` (if it exists) or whatever the equivalent is. What fields move, which are lost, which are duplicated.

**Flow D — Claim creation.**
All entry points: claim wizard UI, demo/seed endpoint that randomly generates claims (mentioned in earlier review), bulk import if any. For each, document what user input is collected, what is auto-filled, what defaults apply.

**Flow E — Claim scrubbing / pre-submission validation.**
What rules fire before EDI generation? Where do they live? What do they check? What blocks submission vs warns vs is silently ignored?

**Flow F — EDI generation.**
From claim record to 837P EDI envelope. Which segments are produced. Which fields source from which entities. Where the "agency_billed" branching happens (architecture doc claims this is on `practice_settings.billing_model`).

**Flow G — Stedi submission.**
How the EDI is sent to Stedi. Auth credentials. Endpoints. Response handling. Where the interchange control number is stored.

**Flow H — 277CA / response ingestion.**
How responses come back from Stedi/PGBA. Where they're parsed. How rejection codes map to claim status. Whether 277CA codes are decoded into human-readable form anywhere.

**Flow I — ERA / 835 ingestion.**
If implemented: how ERAs are retrieved, parsed, matched to claims, posted as payments.

**Flow J — Tenant context propagation.**
How does the application know which organization a request belongs to? Trace from Passport.js login through session through `req.user` through any middleware that uses `organizationId`. Which tables are filtered by `organization_id` in queries today, and which are not.

For each flow, mark every step as one of: `IMPLEMENTED`, `STUBBED` (returns hardcoded/random data), `PARTIALLY_IMPLEMENTED`, `NOT_IMPLEMENTED`, `UNVERIFIED`.

## Section 3 — Module boundaries (`03-module-boundaries.md`)

ClaimShield was structured as two modules in one codebase: Billing and Intake. Document the actual current state of this separation.

- Path-based routing: confirm `/billing/*` and `/intake/*` route prefixes (or note actual structure if different)
- Sidebar / navigation components: which file(s), what they show per role
- Role-based access control: what's actually enforced vs what's defined in the schema. Run a search across `server/` for `req.user.role`, role checks, role middleware. Document every check found.
- Module selector / landing page: does it exist?
- The "Intake Referrals" view in the Billing module: is it implemented? What does it show?
- Cross-module data dependencies: which billing features read from intake tables, which intake features read from billing tables

Produce a table of every page/feature mapped to: which module it belongs to, which roles can access it, whether RBAC is actually enforced for it.

## Method

You may need to run read-only SQL queries to produce the schema inventory. Use a read-only database role if available, or be careful to use only SELECT statements. Save any non-trivial SQL queries used into `docs/architecture/system-audit/_queries/` for reproducibility.

Do not write any data. Do not run any migration. If a tool or operation tries to mutate state, abort.

Use grep, ripgrep, or your preferred search to enumerate routes, env vars, role checks, and TODO markers. Cite file paths and line numbers.

For data flow narratives, do not invent intermediate steps. If you cannot trace a path through the code, say so and stop tracing — partial accurate documentation is more valuable than complete invented documentation.

Mermaid diagrams should use the `flowchart` or `sequenceDiagram` syntax for clarity. Keep diagrams to a screen-sized scope; if a flow is longer, break into sub-diagrams.

## Verification

After producing all 12 files plus the README:

2. Run a word count per file. None should be empty. If a section has no content (e.g., no background jobs exist), the file should explicitly say "No background jobs are currently implemented" rather than being empty.
3. Spot-check three random claims in the docs by opening the cited file/line and confirming the claim is accurate.
4. Confirm no schema changes were made: `SELECT COUNT(*) FROM information_schema.tables WHERE table_schema = 'public'` should match the count from the start of the audit.
5. Confirm no rows were modified in any table that has an `updated_at` column: spot-check three tables and verify no `updated_at` is more recent than the start of the audit.
