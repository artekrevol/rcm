TASK: Add Referring Provider Loop 2310A — table-based, reusable across claims/auths

CONTEXT
VA CCN claims require Loop 2310A (Referring Provider). Without it,
TriWest cannot link the claim to the authorizing VA provider, causing
claim-level rejections.

The same VA provider often refers multiple patients to the same agency.
Rachel confirmed in our last call that Jessica Capistrano is the
referring provider for both test patients. Storing referring providers
as free-text fields per-auth would force re-entry on every record and
introduce NPI transcription errors. The right model is a reusable
Referring Providers directory.

The recently-merged segment builder (server/services/edi/segments.ts)
already exports a typed NM1() helper. The generator change for this
ticket is a single helper call — the heavy lifting is in the data
model and UI.

==== STEP 1: Database ====

Create a new table:

  referring_providers
    id              UUID PK
    tenant_id       UUID FK NOT NULL
    first_name      VARCHAR(50) NOT NULL
    last_name       VARCHAR(50) NOT NULL
    npi             VARCHAR(10) NOT NULL  -- enforce 10-digit numeric
    provider_type   CHAR(1)  NOT NULL DEFAULT '1'  -- '1'=individual, '2'=organization
    notes           TEXT NULL  -- optional, e.g. "SF VAMC PACT MP"
    created_at      TIMESTAMP NOT NULL
    updated_at      TIMESTAMP NOT NULL

  UNIQUE(tenant_id, npi)  -- prevent duplicate provider entries within a tenant

Add foreign-key columns:

  prior_authorizations
    referring_provider_id  UUID FK NULL → referring_providers.id

  claims
    referring_provider_id  UUID FK NULL → referring_providers.id
    -- nullable; resolution at EDI-generation time falls back to the
    -- linked auth's referring provider if claim's own field is null

Use the seeder migration pattern for ALTER TABLEs that's already
established in the codebase.

==== STEP 2: NPI validation ====

Create server/services/validation/npi.ts:

  /**
   * Validates an NPI per the CMS algorithm:
   *  - Must be exactly 10 digits, all numeric
   *  - Last digit is a Luhn check digit computed against the
   *    9-digit prefix '80840' + the first 9 digits of the NPI
   *  - See https://www.cms.gov/Regulations-and-Guidance/Administrative-Simplification/NationalProvIdentStand/Downloads/NPIcheckdigit.pdf
   */
  export function isValidNpi(npi: string): boolean
  export function validateNpiOrThrow(npi: string): void  // throws with descriptive message

Use this on:
  - The referring_providers form on save
  - The prior_authorizations save (when a referring_provider_id is set,
    the agent should not need to re-validate, but the FK relationship
    means the underlying NPI is already validated)
  - At EDI generation, defensively (throws if somehow an invalid NPI
    made it into the database)

Add unit tests covering: valid NPIs (use 1184288680 — Chajinel's NPI as
a known-good test value), invalid length, non-numeric chars, failed
Luhn check, empty string, null.

==== STEP 3: UI — Referring Providers directory ====

Add a new page under Practice settings:
  /billing/settings/referring-providers

Table view with columns: Name, NPI, Type, Notes, Actions (Edit, Delete).
Create/Edit dialog with fields:
  - First Name (required, max 50)
  - Last Name (required, max 50)
  - NPI (required, 10 digits, real-time Luhn validation)
  - Provider Type (radio: Individual / Organization, default Individual)
  - Notes (optional, multi-line, max 500)

Disable Save until the NPI passes validation.

==== STEP 4: UI — Prior Auth form integration ====

On the Prior Auth create/edit form, add a "Referring Provider" section
with:
  - A searchable dropdown (autocomplete by name or NPI) populated from
    the referring_providers table for the current tenant
  - A "+ Add new referring provider" button that opens the create
    dialog inline (returns the new ID and selects it on close)

For VA CCN payers specifically, this field is REQUIRED. Add validation
to the auth save: if the linked payer's claim_filing_indicator is 'VA'
(or the payer is identified as VA CCN by some other config field),
referring_provider_id must be non-null.

For non-VA payers, the field is optional.

==== STEP 5: Generator change ====

In server/services/edi-generator.ts, add the Referring Provider loop
after the HI (diagnosis) emission and before the Loop 2310B (Rendering
Provider) emission.

Resolution logic at the top of the function:

  const referringProvider =
    claim.referring_provider_id
      ? await getReferringProvider(claim.referring_provider_id)
      : auth?.referring_provider_id
        ? await getReferringProvider(auth.referring_provider_id)
        : null;

Emission:

  if (referringProvider) {
    validateNpiOrThrow(referringProvider.npi);
    edi += NM1({
      entityIdCode: 'DN',
      entityTypeQualifier: referringProvider.provider_type as '1' | '2',
      lastOrOrgName: referringProvider.last_name,
      firstName: referringProvider.first_name,
      idQualifier: 'XX',
      idCode: referringProvider.npi,
    });
  }

That's it. The typed helper handles all element positioning. No raw
asterisks anywhere.

==== STEP 7: Tests ====

Generator tests (server/services/edi-generator.test.ts):

  - Case: claim has referring_provider_id → NM1*DN appears with
    correct elements at correct positions (use the parser, not
    substring matching):
      const dn = segments.find(s => s.id === 'NM1' && s.elements[1] === 'DN');
      expect(dn.elements[3]).toBe('Capistrano');
      expect(dn.elements[4]).toBe('Jessica');
      expect(dn.elements[8]).toBe('XX');
      expect(dn.elements[9]).toBe('<10-digit NPI>');

  - Case: claim has no referring_provider_id, auth has one → NM1*DN
    appears (claim falls back to auth)

  - Case: neither claim nor auth has referring provider → no NM1*DN
    in output (must not crash, must not emit empty segment)

  - Case: provider_type='2' → NM1*DN*2 emitted (organization)

  - Case: NM1*DN appears AFTER HI segment and BEFORE NM1*82 segment
    (verify order via segment index in parsed output)

NPI validation tests (server/services/validation/npi.test.ts):
  - Valid 10-digit Luhn-passing NPI accepted
  - 9-digit, 11-digit rejected
  - Non-numeric rejected
  - Failed Luhn check rejected

Auth validation tests:
  - Saving a VA CCN auth without referring_provider_id throws
  - Saving a non-VA auth without referring_provider_id succeeds

==== OUT OF SCOPE ====

- Do NOT add Operating Provider (Loop 2310A variants) or Service
  Facility Location (Loop 2310C/D). Only Referring Provider (DN
  qualifier) is needed.
- Do NOT add multi-referring-provider support per claim. One referring
  provider per claim is the spec.
- Do NOT change the resolveVeteranId function or the NM1*IL emission.
- Do NOT touch Rendering Provider (Loop 2310B) — Rachel confirmed
  agency NPI is correct.
- Do NOT bake the validation rule "VA CCN payers require referring
  provider" into the EDI generator. It belongs in the auth save logic.
  The generator should emit cleanly whether the field is populated or
  not; missing-field validation is upstream.
