# ClaimShield — Phase 3 Sprint 1a: Safety Prerequisites + Tier 1 Wire-In

This sprint follows directly from Phase 3 Sprint 0 (completed 2026-05-03, see `docs/architecture/sprint0-audit-report.md` and `docs/architecture/migration-state.md`). It closes two safety/drift gaps surfaced by Sprint 0 and wires the Tier 1 validator into the live rules engine.

This sprint is intentionally conservative. Daniela starts on the platform Tuesday. Sprint 1a must not change anything she sees. The only behavior change a user could possibly observe is that a structurally-broken claim now returns Tier 1 findings before EDI generation instead of being silently passed to Stedi.

By end of sprint, the new code paths shipped in Sprint 0 are safe to extend with write helpers in future sprints.

## Anchors from Sprint 0

These facts are confirmed in `docs/architecture/sprint0-audit-report.md` (cited as `audit:§N`):

- 6 tenant-scoped tables have RLS + FORCE RLS (audit:§1, §2.5a–c).
- Each has 2 policies: `tenant_isolation` (USING only) and `service_role_bypass` (USING + WITH CHECK) (audit:§2.5b–c).
- The `tenant_isolation` policies are missing `WITH CHECK` clauses. DDL to fix this is **already drafted** in `docs/architecture/migration-state.md §3.1`. Sprint 1a applies that DDL.
- `claimshield_app_role` is the runtime role for tenant-scoped queries (audit:§2.5f). `withTenantTx` issues `SET LOCAL ROLE` per request.
- Helper service layer at `server/services/practice-profile-helpers.ts` (audit:§2.6). Names: `getActivePracticeProfile`, `getEnrolledPayers`, `getActiveProviders`, `getProviderPayerParticipation`, `getPatientCoverages`, `getClaimProviderAssignments`.
- Tier 1 validator at `server/services/rules-engine/tier1-structural-integrity.ts` (audit:§2.7). Pure function `validateTier1Structural(input)` returning `Tier1Finding[]`. Eight rules, codes T1-001 through T1-008.
- Tier 1 is **not yet wired** into `server/services/rules-engine.ts`'s `evaluateClaim` (audit:§2.7).
- Drizzle definition of `organizations` declares 4 columns; the DB has 7 (audit:§3 — `onboarding_dismissed_at, contact_email, status` exist in DB, missing from Drizzle).
- App connects as `postgres` (superuser); RLS only enforced via `withTenantTx` → `SET LOCAL ROLE claimshield_app_role` (audit:§2.5f).

## What this sprint is

1. Pre-flight: snapshot, baseline verification.
2. Apply WITH CHECK DDL (closes Sprint 1 prerequisite).
3. Fix Drizzle drift on `organizations` (adds the 3 missing column declarations).
4. Wire `validateTier1Structural` into `evaluateClaim` so Tier 1 findings are returned alongside legacy rule evaluation.
5. Investigate (read-only) whether the EDI generator path calls `evaluateClaim` before claim generation, and document the trace.
6. Verification, smoke test, audit report.

- No changes to the claim wizard, payer dropdown, voice persona builder, intake forms, or any frontend component.
- No changes to seeded data (Chajinel's `organization_practice_profiles` row stays, demo-org-001's 2 enrollments stay at `enrollment_status='pending'`).
- No write helpers added to the helper service layer.
- No changes to `server/services/edi-generator.ts` itself — only investigation.
- No changes to `replit.md` or any documentation outside `docs/architecture/`.
- No production deploy.

If a step seems to require touching a frontend component, Daniela's UX, or the EDI generator's logic, stop and flag.

## Step 2 — Apply WITH CHECK DDL

### 2a. Locate the DDL

The DDL is drafted in `docs/architecture/migration-state.md §3.1`. Read that section, copy the DDL verbatim into `docs/architecture/sprint1a-snapshots/sprint1a-with-check.sql`. Six `ALTER POLICY` statements expected — one per tenant-scoped table:

- `organization_practice_profiles`
- `practice_payer_enrollments`
- `provider_practice_relationships`
- `provider_payer_relationships`
- `patient_insurance_enrollments`
- `claim_provider_assignments`

If `migration-state.md §3.1` is missing or its DDL doesn't match the expected shape, stop and flag — do not improvise the DDL.

The expected pattern is:

```sql
ALTER POLICY tenant_isolation ON <table_name>
  USING (organization_id = current_setting('app.current_organization_id', true))
  WITH CHECK (organization_id = current_setting('app.current_organization_id', true));
```

Per the GUC name actually used in Sprint 0 (`current_setting('app.current_organization_id', true)` — confirm against `docs/architecture/sprint0-audit-report.md §2.5b` before applying; one Sprint 0 doc references `app.current_org_id` and the other references `app.current_organization_id`, so verify the live setting name via `pg_policies` first).

### 2b. Verify the actual GUC name in use

Run:

```sql
SELECT polname, pg_get_expr(polqual, polrelid) AS using_expr
FROM pg_policy
WHERE polname = 'tenant_isolation';
```

This returns the exact `current_setting(...)` argument used in the existing USING clause. The new WITH CHECK must use the **identical** argument string. Mismatch will silently break isolation.

### 2c. Apply the DDL

Run the six `ALTER POLICY` statements in a single transaction.

### 2d. Verify

```sql
SELECT polname, polrelid::regclass AS table_name,
       pg_get_expr(polqual, polrelid) AS using_expr,
       pg_get_expr(polwithcheck, polrelid) AS with_check_expr
FROM pg_policy
WHERE polname = 'tenant_isolation'
ORDER BY polrelid::regclass::text;
```

Expected: 6 rows, each with both `using_expr` and `with_check_expr` populated and identical to each other. Save output to the audit report §2.

### 2e. Re-run isolation script

```bash
npx tsx scripts/verify-tenant-isolation.ts
```

Still 12/12. The WITH CHECK addition does not affect SELECT — these tests must remain unchanged.

### 4b. Read Tier 1 validator's input/output

Open `server/services/rules-engine/tier1-structural-integrity.ts`. Identify:

- `validateTier1Structural` input type — is it the same Claim shape as `evaluateClaim`'s input, or a different intermediate shape?
- `Tier1Finding` shape: code, severity, message, fieldPath, suggestedFix (per audit:§2.7).

If the input shapes differ, write a thin adapter function in `server/services/rules-engine/tier1-adapter.ts` that maps the legacy claim shape to whatever Tier 1 expects. Do NOT modify Tier 1's signature — its tests pass and we don't want to invalidate them.

### 4c. Wire Tier 1 at the start of `evaluateClaim`

The integration pattern:

```typescript
// At the top of evaluateClaim, before any legacy rule evaluation:
const tier1Findings = validateTier1Structural(adaptToTier1(claim));

// Map Tier1Finding → the shape evaluateClaim returns
const mappedTier1 = tier1Findings.map(f => ({
  code: f.code,
  severity: f.severity,        // 'block' from Tier 1 → whatever 'block' equivalent is in legacy shape
  message: f.message,
  fieldPath: f.fieldPath,
  source: 'tier1-structural',  // tag so callers can distinguish if needed
  // ...other legacy-shape fields filled with defaults
}));

const blockingTier1 = mappedTier1.filter(f => f.severity === 'block');

if (blockingTier1.length > 0) {
  // Short-circuit. Legacy rules cannot evaluate correctly on a structurally broken claim.
  return {
    findings: mappedTier1,
    // ...whatever else the legacy return shape needs (score, status, etc.)
    shortCircuited: true,
    shortCircuitReason: 'tier1-structural-blocking',
  };
}

// If Tier 1 passes, continue with existing legacy rule evaluation.
// Legacy findings get added to mappedTier1 (which is empty or non-blocking warnings only).
const legacyFindings = await runExistingRules(claim);
return {
  findings: [...mappedTier1, ...legacyFindings],
  // ...
};
```

The exact shape of the return depends on what `evaluateClaim` already returns. Match that shape; do not invent new fields except `source`/`shortCircuited` if they're useful additions.

### 4d. Add a test

Create `server/services/rules-engine.test.ts` (or extend if it exists):

- A claim with no service lines → `evaluateClaim` returns Tier 1 finding T1-003, short-circuited.
- A claim with all Tier 1 rules satisfied → `evaluateClaim` proceeds to legacy rules and returns whatever they return.
- A claim with Tier 1 satisfied but a legacy rule violation → `evaluateClaim` returns the legacy finding, no Tier 1 findings.

Use the same `tsx`-runnable assertion pattern Sprint 0 used. Run: `npx tsx server/services/rules-engine.test.ts`.

If the existing rules engine has integration tests that mock or stub specific behavior, the new wiring may break them — investigate and fix any broken tests, but only if the fix is mechanical. If a test breakage indicates a real semantic change, stop and flag.
