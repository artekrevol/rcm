Build a PDF generation endpoint:
POST /api/billing/claims/:id/pdf
This endpoint:

Fetches the full claim record by ID
Fetches the linked patient record
Fetches the linked provider record via claims.provider_id
Fetches practice settings
Fetches payer name — use payers.name if claims.payer_id exists, fall back to claims.payer text field for seeded data
Generates the PDF using @react-pdf/renderer (server-side or client-side — see note below)
Stores the PDF and updates claims.pdf_url
Returns the PDF as a downloadable response

Note on server vs client rendering: @react-pdf/renderer can run on both server and client. Given this is a Replit Express + React setup, the cleanest approach is client-side PDF generation — generate the PDF in the browser and trigger a download directly. This avoids file storage complexity for Phase 1. The endpoint still records that a PDF was generated by updating claims.pdf_url to a placeholder value like 'generated:[timestamp]' and claims.status to 'exported', but the actual PDF bytes are generated and downloaded client-side.

Watch for one specific issue: @react-pdf/renderer uses its own layout engine and doesn't support all CSS. The most common failure is using display: flex without importing View from @react-pdf/renderer, or trying to use HTML elements (div, p) instead of the PDF-specific ones (View, Text, Page, Document). If Replit generates a PDF component that uses standard HTML, it won't render. The component must use only @react-pdf/renderer primitives.
