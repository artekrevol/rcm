Prompt C — Plan Products + Delegated Entities (Conditional Form Activation)
Context
C0 shipped the activation foundation: practice_payer_enrollments, field_definitions, the getActivatedFieldsForContext resolver. C0 deliberately stopped short of adding any conditional fields — every seeded field has always_required = TRUE and activated_by_rule_kinds = []. Universal baseline only.
Prompt C populates the conditional layer. It adds:

A structured plan_products reference table replacing free-form JSONB arrays
A delegated_entities table for IPAs / medical groups
4 new conditional fields in field_definitions: patient_plan_product, patient_pcp_id, patient_pcp_referral_id, patient_delegated_entity_id
The wiring from rules → fields, so when an extraction item with rule_kind = 'referrals' is approved for a payer + plan_product, the resolver returns patient_pcp_referral_id as activated
The patient form (and only the patient form — claim wizard wiring is a smaller follow-up) reading from the resolver and rendering fields conditionally

Acceptance: the Screen A → B → C demo works end-to-end with real data.
What Prompt C delivers

plan_products reference table seeded with the structured plan-product taxonomy.
delegated_entities table seeded with placeholder IPA records (real data populated by separate ingestion).
4 new field_definitions rows with activated_by_rule_kinds arrays populated.
manual_extraction_items schema gains a structured applies_to_plan_products: VARCHAR[] column (replacing JSONB usage where applicable) — backwards compatible with existing JSONB array reads.
Patient form rewired to call GET /api/practice/activated-fields on payer change and render the returned field set.
UHC Capitation/Delegation supplement ingested as a supplement document_type, with at least 1 referrals rule and 1 delegation rule approved in the corpus, providing real demo data for the Screen B → C transition.

T4 — Resolver enhancement
Update getActivatedFieldsForContext from C0 to handle the corpus query path that was a stub. Specifically:

Given (organizationId, payerId, planProductCode, delegatedEntityId, serviceDate), query approved extraction items where:

The source document belongs to this payer
The extraction item's applies_to_plan_products includes the planProductCode or ["all"]
The service date falls within effective_start/effective_end (effective_end NULL = currently in effect)


Collect distinct rule_kind values from matching items.
For each field_definition where activated_by_rule_kinds overlaps with the collected rule kinds, add to the activated set.
Return.

Edge case: if planProductCode is NULL but the payer has any rules in the corpus, return patient_plan_product as activated (the resolver tells the form "you need to pick a plan product before I can tell you what else is needed"). This is the chained-disclosure pattern — picking a plan_product re-fires the resolver and reveals the next layer.
T5 — Patient form rewiring
Patient creation/edit form (/clients/new and /clients/:id/edit):

On payer field change → fire GET /api/practice/activated-fields?payerId=X → render the returned field set.
If patient_plan_product is activated → show plan product dropdown populated from payer_supported_plan_products.
On plan_product change → re-fire the resolver with planProductCode set → render the (possibly expanded) field set.
If patient_delegated_entity_id is activated → show IPA dropdown populated from payer_delegated_entities filtered by (payer, plan_product, state).
If patient_pcp_id and patient_pcp_referral_id are activated → show PCP and referral input fields with helper text: "Required for HMO plans. Missing PCP referral may result in claim denial."

Animation/UX: when the form expands or contracts, fade the new/removed fields with a 200ms transition. The visual continuity is the demo's punchline.
Add a "Routing Preview" pane at the bottom of the form showing what the engine knows: "Claims for this patient will be routed to: [payer or delegated entity]." When delegated_entity is selected with a claims_payer_id_override, this pane updates accordingly.
Wire this only into the patient form. Claim wizard wiring is a separate prompt — keep C focused.
T6 — Ingest UHC Capitation/Delegation supplement (real demo data)
Manual operation, scripted:

Download the 2026 UHC Administrative Guide PDF.
Extract pages 182–251 (the Capitation/Delegation supplement section) as a separate PDF — this fits well under the 100-page Claude limit.
Upload as a new payer_source_documents row with document_type = 'supplement', parent_document_id pointing to the existing UHC admin_guide row (create the parent first if it doesn't exist).
Run extraction.
Review and approve at least one rule in each of these kinds: referrals, prior_auth, appeals. Set applies_to_plan_products = ["ma_hmo", "ma_dsnp"] on referrals rules.
Confirm the Patient form, when payer is set to UHC and plan_product is set to "Medicare Advantage HMO," now shows the PCP and referral fields conditionally.

This is the real-data backbone of the demo. Without it, the form has nothing to activate against.
