# Multi-Tenancy Refactor — Intake Module

## Goal

Refactor the intake module from single-tenant (constants files) to 
fully multi-tenant (per-org database tables). After this work, adding 
a new client means inserting rows into a defined set of tables — 
no code changes, no redeploys.

Currently only two organizations exist:
- demo-org-001 (ClaimShield Demo Practice — currently holds the 
  seeded Caritas flow as a demo artifact)
- chajinel-org-001 (Chajinel Clinic — Daniela's clinic, no intake 
  flow yet)

This refactor will:
1. Create caritas-org-001 as a proper organization
2. Move the existing Caritas flow from demo-org-001 to caritas-org-001
3. Add a new Chajinel intake flow under chajinel-org-001
4. Make the orchestrator and step executor entirely org-agnostic
5. Eliminate caritas-constants.ts and any other hardcoded per-client 
   logic

## Critical instructions

Branch off main. Do NOT deploy to production until Abeer reviews 
staging/preview and signs off. All work happens on a feature branch.

If you hit a blocker (missing schema, env var, third-party access 
issue), STOP and tell Abeer rather than working around it.

After each phase, give a verification report before moving to the 
next phase. Do not skip phases.

## Phase A — Schema audit and extensions

### A.1 — Audit existing schema

First, run these queries and report results back to Abeer:

SELECT table_name FROM information_schema.tables 
WHERE table_schema = 'public' 
  AND table_name SIMILAR TO '%(flow|run|lock|template|service|payer|persona|provider|source|org)%'
ORDER BY table_name;

For each existing flow-related table (flows, flow_steps, flow_runs, 
flow_run_events, comm_locks), run:

SELECT column_name, data_type, is_nullable, column_default 
FROM information_schema.columns 
WHERE table_name = '<table_name>' 
ORDER BY ordinal_position;

This tells us what already exists vs. needs to be added.

### A.2 — Extend existing tables

Based on the audit, ADD these columns if missing (use ALTER TABLE 
ADD COLUMN IF NOT EXISTS, do not drop or recreate existing tables):

flows:
- org_id TEXT NOT NULL (reference organizations(id))
- version INTEGER NOT NULL DEFAULT 1
- updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()

flow_steps:
- condition JSONB (nullable; absent means always run)
- max_attempts INTEGER NOT NULL DEFAULT 3
- success_criteria JSONB (nullable; future use for advanced step 
  completion logic)

flow_runs:
- org_id TEXT NOT NULL (reference organizations(id))
- attempt_count INTEGER NOT NULL DEFAULT 0
- failure_reason TEXT
- failed_at TIMESTAMPTZ
- completed_at TIMESTAMPTZ

comm_locks:
- released_at TIMESTAMPTZ (nullable until released)
- released_reason TEXT
- expires_at TIMESTAMPTZ NOT NULL (for orphan lock cleanup)

Add indexes if missing:
- CREATE INDEX IF NOT EXISTS idx_flows_org_active 
  ON flows(org_id, is_active);
- CREATE INDEX IF NOT EXISTS idx_flow_runs_org 
  ON flow_runs(org_id, status);
- CREATE INDEX IF NOT EXISTS idx_flow_runs_due 
  ON flow_runs(status, next_action_at) WHERE status = 'running';

## Phase C — Refactor step executor to be org-agnostic

### C.1 — Create org-context service

Create server/services/org-context.ts:

- Function: getOrgContext(orgId: string): Promise<OrgContext>
- Fetches and returns: { org_id, templates, service_types, payers, 
  personas, lead_sources, providers }
- templates and personas are keyed by template_key/persona_key for 
  easy lookup
- Cache results in-memory for 60 seconds per org_id (avoids hammering 
  the DB on hot paths). Invalidate cache when admin updates an org's 
  config (later sprint).

### C.2 — Refactor flow-step-executor

In server/services/flow-step-executor.ts:

- Remove ALL imports from server/config/caritas-constants.ts
- Every step execution function takes (run, step, orgCtx) parameters
- Templates fetched via orgCtx.templates[step.template_key]
- Voice personas fetched via orgCtx.personas[step.config.persona_key]
- Service types fetched via orgCtx.service_types
- Payer mappings fetched via orgCtx.payers
- Providers fetched via orgCtx.providers

Implement these step types fully:

1. wait — already exists, keep as is (just delays flow advancement)

2. sms_message:
   - Get template from orgCtx.templates[step.template_key]
   - Render template body with run/lead variables ({first_name}, etc.)
   - Send via Twilio
   - Emit flow_run_event with type 'sms_sent'
   - Advance to next step

3. voice_call:
   - Get persona from orgCtx.personas[step.config.persona_key]
   - Acquire comm_lock with channel='voice', expires in 30 min
   - POST to Vapi /call/phone with assistantId from persona, 
     metadata { leadId, flowRunId, lockId, orgId }
   - Do NOT include 'server' property at top level (this is the 
     bug from yesterday — webhook is configured at assistant level 
     in Vapi dashboard)
   - Emit event 'voice_call_initiated' with vapi_call_id
   - Do NOT advance immediately — webhook handler advances after 
     end-of-call-report received

4. email_message:
   - Get template from orgCtx.templates with channel='email'
   - Render subject + body with variables
   - Send via Gmail/SendGrid (whichever is currently wired)
   - Emit event 'email_sent'
   - Advance to next step

5. vob_check:
   - Use lead's insurance info + orgCtx.payers to map carrier to 
     payer_id
   - Call Stedi 270/271 API
   - Store result on lead record (vob_status, eligibility_data)
   - Emit event 'vob_completed' with result summary
   - Advance to next step (or failure branch if VOB returns hard error)

6. provider_match:
   - Filter orgCtx.providers by:
     - service_types contains lead.service_type_requested (if set)
     - languages contains lead.language_preference (if set)
     - availability supports lead.urgency (parse availability JSON)
   - If no matches, emit 'no_provider_match' event and mark run 
     for failure handoff
   - If matches, pick best (round-robin, capacity, score — start 
     with simple round-robin)
   - UPDATE leads SET matched_provider_id = ? WHERE id = ?
   - Emit event 'provider_matched' with provider details
   - Advance to next step

7. appointment_schedule:
   - Two modes: 'auto' (future) and 'manual_handoff' (now)
   - For manual_handoff: query org admins (users with role 'admin' 
     for the org), send notification SMS + email to each: "New lead 
     {first_name} matched to {provider_name}. Please schedule 
     appointment at https://claimshield.health/intake/leads/{id}"
   - Emit event 'appointment_handoff_sent'
   - Advance to next step

8. webhook (basic for v1):
   - POST to step.config.url with run+lead context
   - Honor step.config.method (default POST), headers, timeout
   - Emit event 'webhook_called' with response status
   - Advance unless response is failure status code

9. conditional — handled at executor level via step.condition 
   (see C.3), not as a separate step type

### C.3 — Implement condition evaluator

Before executing each step, check step.condition. If condition is 
not null, evaluate against run+lead context. If false, mark step 
as 'skipped' in flow_run_events and advance to next step without 
executing.

Condition shape:
{ "field": "lead.payment_method", "operator": "in", "value": ["insurance", "medicare"] }

Supported operators:
- eq, neq
- in, not_in
- exists, not_exists
- gt, gte, lt, lte (numeric)
- contains (string contains)

Field path resolution: lead.* fields come from leads table, run.* 
from flow_runs, run.metadata.* from flow_runs.metadata JSONB.

### C.4 — Implement retry cap and failure handling

In step executor:
- On step failure (exception, non-2xx response, etc.):
  - Increment flow_runs.attempt_count
  - If attempt_count < max_attempts (from flow_steps.max_attempts):
    - Schedule retry: next_action_at = NOW() + backoff
    - Backoff: attempt 1 = +5min, attempt 2 = +15min, attempt 3 = 
      mark failed
  - If attempt_count >= max_attempts:
    - UPDATE flow_runs SET status='failed', failure_reason=<error>, 
      failed_at=NOW()
    - Emit event 'step_failed_terminal'
    - Stop processing this run

In Flow Inspector UI:
- Collapse repeated step_started/step_failed pairs for the same 
  step into a single row: "Step X: <type> — failed after N attempts: 
  <reason>"
- Show "Retry from this step" button when run is in 'failed' status

### C.5 — Refactor orchestrator

In server/services/orchestrator.ts:
- Query for due runs: include lead's org_id in the join
- Pass org_id through to step executor calls

### C.6 — Webhook handler

In the Vapi webhook endpoint:
- Verify signature/secret
- Look up flow_run by metadata.flowRunId
- Release the comm_lock (set released_at, released_reason='vapi_webhook')
- Run server-side Claude Haiku transcript extraction (existing 
  behavior)
- UPDATE flow_runs to advance current_step_order, set 
  next_action_at to next step's scheduled time
- If next step has delay_minutes, next_action_at = NOW() + 
  delay_minutes * 60 seconds
- Emit event 'voice_call_completed' with extraction results

This is critical — yesterday's bug was that the webhook didn't 
advance the run, so the orchestrator kept re-firing the call step. 
Make sure step advancement is atomic and reliable.
