This sprint cleans up everything Daniela (Chajinel owner) and Rachel (her biller) flagged during the May 1 live walkthrough. Rachel will run a live test claim through ClaimShield on Tuesday. ## Section 1 — Patient form: middle name field

**Problem:** The patient creation form has First Name and Last Name only. Daniela: "most Americans have two names. So, first name, middle name." The 837P NM1*IL segment supports a middle name and the absence of one occasionally causes match issues on the payer side.

**Implementation:**
- Add `middle_name VARCHAR(50) NULL` to `patients` table via Drizzle migration.
- Update the patient creation/edit form: `First Name (Required)` | `Middle Name (Optional)` | `Last Name (Required)`. Middle name field is optional and stays empty by default.
- Update patient display components (claim wizard patient picker, patient list, claim summary PDF) to show middle initial where relevant: "John D. Smith" if middle name present, "John Smith" if not.
- Update the 837P generator: in Loop 2010BA / NM1*IL, populate the middle name element (NM105) when present. Confirm element ordering: NM1*IL*1*[last]*[first]*[middle initial]*...

**Verification:**
1. Create a new patient with First/Middle/Last filled. Confirm middle name persists after save.
2. Edit an existing patient and add a middle name. Confirm migration didn't drop existing patient records.
3. Generate 837P for a claim attached to a patient with middle name. Confirm NM1*IL segment shows the middle name.
4. Confirm display strings render correctly across patient list, claim detail, and claim summary PDF.

## Section 2 — Service line: per-hour rate input with auto-conversion

**Problem:** The service line currently asks for `Rate per Unit` ($14.25/unit). VA gives Daniela a `Rate per Hour` ($57/hour). She doesn't think in 15-minute units. The wizard should accept the input in the unit she actually uses and do the conversion behind the scenes.

**Implementation:**
- Add `rate_input_mode VARCHAR(20) DEFAULT 'per_unit'` to `payers` table. Allowed values: `'per_unit'`, `'per_hour'`.
- For the VA Community Care payer record (created in Section 7), set `rate_input_mode = 'per_hour'`.
- In the service line UI, when the claim's payer has `rate_input_mode = 'per_hour'`:
  - Replace the `Rate ($/unit)` input with `Hourly Rate ($/hour)`.
  - Below the input, show a live calculation strip: `$57.00/hour ÷ 4 units/hour = $14.25/unit · 8 hours × 4 = 32 units · 32 × $14.25 = $456.00`.
  - The strip recalculates as the user changes hours or hourly rate.
  - The hours field (already exists) drives the unit count: `units = hours × 4` for 15-minute unit codes.
- For EDI generation (837P SV1 segments), continue serializing as `units × unit_rate` — HIPAA 5010 requires per-unit pricing in SV102/SV103. The conversion happens at submit time, not in the database. Internally compute `unit_rate = hourly_rate / 4` and `units = hours × 4`.
- Add a `unit_minutes INT DEFAULT 15` column to the service codes table so 30-minute or hourly codes also work correctly with this conversion. For 15-minute codes (G0156, G0299, etc.), units/hour = 4. For 60-minute codes, units/hour = 1. Conversion math reads from this field.

**Verification:**
1. Open a claim for the Chajinel tenant with VA Community Care as payer. Add a service line with code G0156 (or whatever 15-min code is loaded). Enter `Hourly Rate = 57` and `Hours = 8`. Confirm the calculation strip shows the math live and the line total is `$456.00`.
2. Change hours to 4. Confirm strip recalculates: `4 × 4 = 16 units · 16 × $14.25 = $228.00`.
3. Submit a test claim through Stedi inspector. Confirm the SV1 segment shows the per-unit rate (14.25) and unit count (32), not the hourly figure.
4. Switch the claim's payer to a `per_unit` payer (e.g. demo tenant payer). Confirm the UI reverts to the original per-unit rate input.
5. Confirm the demo tenant's existing claims and rate inputs are untouched.

## Section 3 — Service line: Clone Line button

**Problem:** Daniela bills monthly. A typical claim has the same service code and rate repeated for 16–20 service dates within the month. Currently every line has to be entered from scratch. Daniela: "you can put, like, here, like, clone, clone your line… you would just have to change the date, so you don't have to keep entering in, entering in, entering in."

**Implementation:**
- Add a `Clone Line` button next to each existing service line in the wizard. Icon + "Clone Line" text label.
- On click: create a new service line with all fields copied from the source line (code, modifier, ICD pointer, hours, rate, dates, etc.) except:
  - The new line's `service_date` is empty (NULL) — force the user to enter the date for the new line.
  - The new line's `service_date` field is auto-focused and visually highlighted (border in your warning/yellow token) so it's obvious that's the field that needs attention.
- The cloned line appears immediately below the source line, not at the bottom of the list.
- Cloning preserves the relative ordering when more lines are added afterwards.

**Verification:**
1. Add a service line with code, hours, rate, date filled in. Click `Clone Line`.
2. Confirm a new line appears immediately below the source with all fields copied except date.
3. Confirm the new line's date field is focused and visually highlighted.
4. Enter a new date. Confirm the highlight clears and the line saves.
5. Clone again from the new line. Confirm the chain works — three lines, three different dates, identical other fields.

## Section 4 — Claim wizard terminology fixes

**Problem:** Daniela and Rachel both stumbled on the labels. Rachel asked what "DX Pointer" meant. Daniela: "people are familiarized with CPT and ICD terminology." The wizard uses correct EDI terminology that doesn't match what working billers see in their day-to-day tools.

**Implementation:** Update only UI labels and helper text. Do not change DB column names, API field names, or EDI generator code.

| Current label | New label |
|---|---|
| `Code` (in service line) | `CPT/HCPCS Code` |
| `DX Pointer` | `ICD Diagnosis Pointer` |
| `Diagnosis` (primary/secondary) | `ICD-10 Diagnosis Code` |
| `Modifier` | `Modifier` *(unchanged — already standard)* |

Add helper tooltips:
- `CPT/HCPCS Code`: "The procedure code for the service performed (e.g. G0156 for home health aide services)."
- `ICD Diagnosis Pointer`: "Points to which ICD-10 diagnosis code on this claim applies to this service line. Pointer 1 = Primary diagnosis."
- `ICD-10 Diagnosis Code`: "The patient's diagnosis code (e.g. Z74.2 for need for assistance at home)."

**Verification:**
1. Open the claim wizard. Confirm all four labels render correctly.
2. Hover each labeled field. Confirm the tooltip text matches the spec above.
3. Confirm no DB or API field names changed — pull a claim record via the API and confirm field names are unchanged from before.

## Section 5 — Required vs Optional explicit field labels

**Problem:** The wizard uses asterisk (`*`) to mark required fields. Rachel got confused mid-flow about which fields she had to fill in. Daniela: "I think you should mark whatever it's required to bill as required, and then put optional if it's optional… so people are not confusing, do I have to submit this part."

**Implementation:**
- Replace asterisks with explicit chip-style labels next to each field label:
  - Required fields: small red chip reading `Required` — use existing destructive/error color token.
  - Optional fields: small gray chip reading `Optional` — use existing muted/secondary color token.
- Apply across: patient creation form, claim wizard (all steps), payer setup, provider setup, user management.
- Required fields keep client-side validation as today; the chip is purely visual.
- For fields where requirement depends on context (e.g. middle name optional, secondary diagnosis optional), the chip reflects the static rule. Conditional logic stays in the validation layer.

**Verification:**
1. Open patient creation form. Confirm all required fields show the `Required` chip and optional fields show `Optional`.
2. Try submitting with required fields blank. Confirm validation still fires correctly.
3. Open the claim wizard. Confirm chips render across every step.
4. Confirm chips are visually distinguishable for colorblind users — chip text alone (`Required` / `Optional`) carries the meaning, not just color.

## Section 6 — Payer-conditional VOB check

**Problem:** Daniela: "we have a referral, and the referral comes already with us." For VA Community Care, the referral document includes the prior authorization, so a VOB step is redundant. The current pre-flight warning fires regardless of payer and creates noise on every Chajinel claim.

**Implementation:**
- Add `requires_vob BOOLEAN DEFAULT TRUE` to `payers` table.
- For the VA Community Care payer (Section 7), set `requires_vob = FALSE`.
- In the claim wizard pre-flight check, only fire the `Patient VOB not verified` warning if `claim.payer.requires_vob = TRUE`.
- Add the `Requires VOB` toggle to the payer setup form so this can be configured per-payer going forward.

**Verification:**
1. Open a Chajinel claim with VA Community Care as payer and no VOB on the patient. Run pre-flight. Confirm the VOB warning does NOT fire.
2. Switch payer to a payer with `requires_vob = TRUE`. Confirm the VOB warning fires.
3. Open payer setup, toggle `Requires VOB` on/off, save. Confirm the change reflects on subsequent pre-flight runs.
