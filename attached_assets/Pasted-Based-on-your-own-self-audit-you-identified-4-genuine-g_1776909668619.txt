### GAP 1 — Ordering Provider Column Name Mismatch

Your audit found that ordering provider data is stored as:
- external_ordering_provider_name
- external_ordering_provider_npi

But the system was built expecting:
- ordering_provider_first_name
- ordering_provider_last_name
- ordering_provider_npi
- ordering_provider_org

This is not just a naming preference — it is a functional 
problem because:

1. The EDI generator looks for ordering_provider_npi 
   to build Loop 2420D (Ordering Provider Name segment).
   If it's stored as external_ordering_provider_npi,
   the loop never fires and the ordering provider 
   is absent from every 837P submission.

2. The CMS-1500 PDF generator looks for specific field 
   names to populate Box 17 (Referring Provider Name) 
   and Box 17b (NPI). If field names don't match, 
   Box 17 is blank on every generated PDF.

3. The claim Review step displays ordering provider 
   using specific field names. If wrong, it shows blank.

Fix:
Add the correct columns alongside existing ones:
ALTER TABLE claims 
  ADD COLUMN IF NOT EXISTS ordering_provider_first_name VARCHAR,
  ADD COLUMN IF NOT EXISTS ordering_provider_last_name VARCHAR,
  ADD COLUMN IF NOT EXISTS ordering_provider_npi VARCHAR,
  ADD COLUMN IF NOT EXISTS ordering_provider_org VARCHAR;

Migrate existing data:
UPDATE claims SET
  ordering_provider_npi = external_ordering_provider_npi,
  ordering_provider_first_name = SPLIT_PART(
    external_ordering_provider_name, ' ', 1),
  ordering_provider_last_name = SPLIT_PART(
    external_ordering_provider_name, ' ', 2)
WHERE external_ordering_provider_npi IS NOT NULL;

Update ALL references in:
- routes.ts (claim wizard save, claim detail GET)
- edi-generator.ts (Loop 2420D)
- generate-cms1500.ts (Box 17/17b)
- claim-wizard.tsx (ordering provider fields)
- claim-detail.tsx (ordering provider display)

Verify end to end:
1. Enter ordering provider NPI in claim wizard
2. Generate EDI — confirm Loop 2420D appears
3. Generate CMS-1500 PDF — confirm Box 17/17b populated
4. View claim detail — confirm ordering provider visible

### GAP 2 — Patient Address JSONB vs Flat Columns

Patient address is stored as a JSONB object in the 
address column instead of flat columns street_address,
city, zip_code.

This causes two problems:

1. The CMS-1500 generator must parse JSONB to get 
   patient address for Box 5. If the JSON structure 
   varies (street vs street1 vs address), parsing fails 
   silently and Box 5 is blank.

2. The New Patient form currently only has a State field
   with placeholder "TX". No street, city, or ZIP fields
   exist. CMS-1500 Box 5 requires all four components.

Fix both:
a) Add flat columns to patients table:
ALTER TABLE patients
  ADD COLUMN IF NOT EXISTS street_address VARCHAR,
  ADD COLUMN IF NOT EXISTS city VARCHAR,
  ADD COLUMN IF NOT EXISTS zip_code VARCHAR;

b) Migrate existing JSONB address data:
UPDATE patients SET
  street_address = address->>'street',
  city = address->>'city',
  zip_code = address->>'zip'
WHERE address IS NOT NULL;

c) Update New Patient form to include:
- Street Address (text, required)
- City (text, required)
- State (dropdown, 50 states, required)
- ZIP Code (text, 5-digit validation, required)

d) Update POST /api/billing/patients to save 
   flat columns in addition to JSONB address for 
   backwards compatibility.

e) Update GET /api/billing/patients/:id to return 
   flat address fields.

f) Update CMS-1500 generator to use flat columns:
   patientAddress: {
     street: patient.street_address,
     city: patient.city,
     state: patient.state,
     zip: patient.zip_code
   }

g) Verify CMS-1500 Box 5 populates correctly 
   after fix.
