# ClaimShield — Phase 3 Sprint 0: Architectural Foundation (v3)

This is sprint 0 of the architectural restructure described in `ClaimShield_Architecture_Restructure_Scope.docx`. v3 supersedes v1 and v2, anchored to the audit at `docs/architecture/system-audit/`.

By the end of this sprint, the new schema and tenant-isolation infrastructure exist alongside the current code. Existing functionality (Daniela demo path, current intake flows, EDI submissions, claim wizard) must continue to work unchanged.

## Anchors from the audit

These facts are now confirmed (`docs/architecture/system-audit/`):

- `organizations.id` is `varchar` (`shared/schema.ts:31`). All new FK columns use TEXT/VARCHAR, not UUID.
- `payers` exists with 238 rows and FKs in place (`01-database-schema.md`, `_queries/04_foreign_keys.tsv`).
- `practice_payer_enrollments` **already exists** with 2 rows and `organization_id` FK (`01-database-schema.md`). Sprint 0 must reconcile, not recreate.
- `practice_profiles`, `organization_practice_profiles`, `provider_practice_relationships`, `provider_payer_relationships`, `patient_insurance_enrollments`, `claim_provider_assignments` do **not** exist — all greenfield.
- `org_types` and `org_type_field_specs` were proposed in earlier multi-tenancy work but **never built** (`01-database-schema.md` Drift section). No deprecation needed.
- `practice_settings.billing_model` does **not** exist; the related column is `frcpb_enrolled` (`01-database-schema.md` Drift section, `routes.ts:3520`). Don't reference billing_model anywhere.
- No RLS, no triggers, no views, no functions, no stored procedures (`01-database-schema.md`). Greenfield for RLS.
- `org_voice_personas` exists with 2 rows (`01-database-schema.md`). Sprint 0 doesn't touch it; sprint 1 refactors persona builder to read from profile.
- Org context resolver is `getOrgId(req)` at `server/routes.ts:153-160`. Handles super_admin impersonation via `req.session.impersonatingOrgId`. The new middleware mirrors this logic — does not reinvent.
- Auth helpers are at `server/auth.ts:231-260`: `requireAuth`, `requireRole(...allowed)`, `requireSuperAdmin`.
- App bootstrap is `server/index.ts` (122 lines). `setupAuth(app)` is at line 78. Tenant-context middleware slots in between auth setup and `registerRoutes(app)`.
- Pool + Drizzle init is `server/db.ts:7,13`, using `DATABASE_URL` against PostgreSQL 16.10.

## What this sprint is

1. New schema, additive — five relationship tables (one already exists, four new), the `practice_profiles` catalog, an `organization_practice_profiles` join table.
2. **Tenant isolation infrastructure**: PostgreSQL Row-Level Security on the new tables, plus an Express middleware + AsyncLocalStorage pattern that establishes per-request tenant context for the new helper layer.
3. One fully-seeded profile: `home_care_agency_personal_care`, used by Chajinel.
4. Helper service layer for profile resolution and relationship querying — these are the only consumers of the tenant-context middleware in sprint 0.
5. Tier 1 (Structural Integrity) of the rules engine, as a standalone validator.
6. A feature flag, default OFF.
7. Migration-state documentation.

## What this sprint is NOT

- No data migration. Existing rows in payers, providers, patients, claims stay where they are.
- No refactor of existing API routes, EDI generator, claim wizard, voice persona builder, or any frontend component.
- No deletion or renaming of existing tables or columns.
- No backfill of FK constraints on `leads`, `calls`, `chat_sessions`, `appointments`, `flows`, `flow_runs`, `denials`, `era_batches`, `activity_logs`, `email_logs` — those are sprint 2.
- No fix to the duplicate FK on `practice_settings` — sprint 2.
- No implementation of Tiers 2–5 of the rules engine.
- No customer-facing UI changes.
- No Caritas onboarding. That's a follow-up sprint.
- No edits to `replit.md` or any documentation file other than the new ones in `docs/architecture/`.

If a step seems to require touching existing code paths to make the demo work, stop and flag it.

## Hard rules

1. Existing tables are not modified. The single exception: ALTER on `practice_payer_enrollments` only if its existing schema is missing columns required by the new architecture, and only as additive ALTER (no drops, no renames).
2. New code paths coexist with old. Don't rip out current `organization_id` patterns or the `verifyOrg`/`requireOrgCtx` pattern.
3. The feature flag `USE_PROFILE_AWARE_QUERIES` is OFF in `.env` after this sprint.
4. RLS policies are enabled on the new tenant-scoped tables from the moment of creation. They are never readable without tenant context set.
5. Connection-pool sticky-context bugs are the highest-risk failure mode. Use the transaction-scoped approach specified in Step 5.
6. If anything in this prompt conflicts with the audit, the audit wins — flag and propose a fix.
7. Take a database snapshot/dump before running any DDL. Save the dump path in the audit report.

### 5d. Tenant-context middleware (`server/middleware/tenant-context.ts`)

The middleware uses AsyncLocalStorage to hold the per-request tenant context plus a transaction-scoped Drizzle instance. Helpers in Step 6 read from this store. Existing routes do not — they continue to use the global `db` from `server/db.ts`.

```typescript
// server/middleware/tenant-context.ts
import { Request, Response, NextFunction } from 'express';
import { AsyncLocalStorage } from 'async_hooks';
import { sql } from 'drizzle-orm';
import { db } from '../db';

type TenantContext = {
  organizationId: string;
  // Transaction-scoped Drizzle instance — pass to helpers in Step 6
  tx: typeof db;
};

const tenantContextStorage = new AsyncLocalStorage<TenantContext>();

/**
 * Mirrors getOrgId(req) at server/routes.ts:153-160.
 * Super-admin uses req.session.impersonatingOrgId; regular users use req.user.organization_id.
 * Returns null if no org context can be determined.
 */
function resolveOrgIdFromRequest(req: Request): string | null {
  const user = req.user as any;
  if (!user) return null;
  if (user.role === 'super_admin') {
    return (req.session as any)?.impersonatingOrgId ?? null;
  }
  return user.organization_id ?? null;
}

/**
 * Express middleware. For authenticated requests with a resolvable org id:
 *   1. Open a Drizzle transaction
 *   2. SET LOCAL app.current_organization_id = <orgId>
 *   3. Run the rest of the request inside an AsyncLocalStorage scope so helpers see tenant context
 *   4. Commit when the response finishes; rollback on error
 *
 * For requests with no auth (login page, static, public webhooks): no-op.
 * For super-admin without an active impersonation: no-op (helpers return empty results).
 */
export function tenantContextMiddleware(req: Request, res: Response, next: NextFunction) {
  const orgId = resolveOrgIdFromRequest(req);

  if (!orgId) {
    return next();
  }

  // Open a transaction and run the request inside it.
  // SET LOCAL is auto-scoped to the transaction — no manual cleanup needed.
  db.transaction(async (tx) => {
    // Parameterized SET — orgId is from the auth session, but defense-in-depth.
    await tx.execute(sql`SET LOCAL app.current_organization_id = ${orgId}`);

    return new Promise<void>((resolve, reject) => {
      // Establish the AsyncLocalStorage scope, then call next() inside it.
      tenantContextStorage.run({ organizationId: orgId, tx }, () => {
        const onFinish = () => {
          res.removeListener('close', onError);
          resolve();
        };
        const onError = (err?: Error) => {
          res.removeListener('finish', onFinish);
          reject(err ?? new Error('Request closed before finish'));
        };
        res.once('finish', onFinish);
        res.once('close', onError);
        next();
      });
    });
  }).catch((err) => {
    // The transaction rolled back. If the response hasn't been sent, send a 500.
    if (!res.headersSent) {
      res.status(500).json({ error: 'Tenant context error', detail: err.message });
    }
  });
}

/**
 * Helpers (Step 6) call this to get the request-scoped Drizzle instance.
 * Returns the global `db` if called outside a request scope (e.g. background jobs).
 */
export function getTenantDb(): typeof db {
  return tenantContextStorage.getStore()?.tx ?? db;
}

export function getCurrentTenantId(): string | null {
  return tenantContextStorage.getStore()?.organizationId ?? null;
}
```

```typescript

// ...

```

The middleware sits AFTER Passport so `req.user` is populated, and BEFORE routes so handlers run inside the tenant context.

### 5e. Connection-pool concern

The transaction approach means each authenticated request holds a connection from the `db` pool for its full duration. Current ClaimShield load (3 orgs, dev DB, low concurrent traffic) is well within pool capacity. If concurrent request load grows, the pool size in `server/db.ts` may need to increase. Document the current pool size in `sprint0-existing-schemas.md`.

If `server/db.ts` reveals an unusual pool config (e.g. a pgbouncer in transaction mode), stop and flag — the transaction-scoped `SET LOCAL` approach assumes a per-connection lifecycle. With pgbouncer transaction mode it still works because LOCAL is transaction-scoped; with pgbouncer statement mode it does NOT work and the whole approach must change.
