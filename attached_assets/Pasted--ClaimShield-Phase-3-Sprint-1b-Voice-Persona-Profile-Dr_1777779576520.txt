# ClaimShield — Phase 3 Sprint 1b: Voice Persona Profile-Driven Composition

This sprint follows Sprint 1a (completed 2026-05-03 — `docs/architecture/sprint1a-audit-report.md` and `docs/architecture/migration-state.md §8`). It refactors the voice persona system-prompt builder so that the "fields to capture" section is composed at runtime from the active practice profile instead of being baked into the static template stored in `org_voice_personas.system_prompt`.

This is the work that unblocks Caritas onboarding. After Sprint 1b lands, adding a new tenant becomes: define a profile (or reuse one) → create the org row → map to profile → create the persona row with greeting/tone/conversation structure (no field list). The field list is rendered at call time from the profile's `intake_field_specs`.

This sprint is intentionally additive and opt-in. Existing personas are not disturbed. Chajinel's persona (currently inactive — `vapi_assistant_id = 'PLACEHOLDER_AWAITING_VAPI_CONFIG'`) is migrated to the new pattern as the proof point. Daniela's Tuesday demo is unaffected (the EDI/billing path doesn't touch voice personas).

## What this sprint is

1. Pre-flight: snapshot, baseline verification.
2. Discovery (read-only): map every code path that loads, customizes, or sends a voice persona system prompt to Vapi.
3. Schema: add `compose_from_profile boolean NOT NULL DEFAULT false` to `org_voice_personas` (additive; default false preserves existing behavior).
4. Drizzle: update `org_voice_personas` definition in `shared/schema.ts` to include the new column.
5. New service: `server/services/voice-persona-builder.ts` exporting `buildAssistantSystemPrompt(orgId)`.
6. Wire the builder into the outbound Vapi call path at `flow-step-executor.ts:566-571`.
7. Migrate Chajinel's persona to the new pattern (set `compose_from_profile=true`, update `system_prompt` to use the placeholder convention).
8. Tests for the builder.
9. Verification, smoke test, audit report, migration-state update.

## What this sprint is NOT

- No changes to the Vapi-side static assistant configuration (the dashboard / Vapi API). The `VAPI_ASSISTANT_ID`-configured prompt is out of scope; sprint 1c handles inbound-side reconciliation.
- No changes to the inbound call flow (the Vapi webhook signature path at `routes.ts:9279`). If discovery in Step 2 surfaces that inbound prompts come from Vapi's static config, document and defer.
- No changes to any persona other than Chajinel's. The second `org_voice_personas` row stays at `compose_from_profile=false` and is byte-identical post-sprint.
- No activation of Chajinel's Vapi assistant (`vapi_assistant_id` stays at the placeholder string).
- No changes to `practice_profiles.intake_field_specs` content.
- No changes to the EDI generator, claim wizard, payer dropdown, billing module, or any frontend component.
- No production deploy.
- No Caritas onboarding (sprint 2+).

If a step seems to require touching Vapi's dashboard, activating an assistant, or modifying the inbound webhook, stop and flag.

## Hard rules

1. Take a database snapshot before any DDL or row update. Save the path in the audit report.
2. The schema change is additive only — one new column with a NOT NULL default. No drops, no renames, no type changes.
3. Existing personas with `compose_from_profile=false` (every row at sprint start) must produce byte-identical assembled system prompts after sprint 1b as before. The builder's no-op path is the contract.
4. The builder must use `withTenantTx` indirectly through the existing helper layer. Do NOT bypass to the global `db` for any tenant-scoped read.
5. If `getActivePracticeProfile(orgId)` returns null (org not mapped to a profile), the builder must fall back to returning `system_prompt` verbatim — even when `compose_from_profile=true`. This fail-safe prevents an unmapped org from sending an empty prompt to Vapi.
6. Standing order: dev only. No production deploy.

## Step 5 — Build `voice-persona-builder.ts`

Create `server/services/voice-persona-builder.ts`. The single export is `buildAssistantSystemPrompt(orgId: string): Promise<string>`.

### 5a. Function contract

```typescript
import { withTenantTx } from '../middleware/tenant-context';
import { db } from '../db';
import { orgVoicePersonas } from '@shared/schema';
import { eq } from 'drizzle-orm';
import { getActivePracticeProfile } from './practice-profile-helpers';

export async function buildAssistantSystemPrompt(orgId: string): Promise<string> {
  // 1. Load the persona row for this org.
  //    Persona is loaded outside withTenantTx because org_voice_personas does NOT
  //    have RLS in sprint 1b — the table predates Phase 3. Keep using global `db`
  //    until persona-table RLS migration in a later sprint (track in migration-state).
  const persona = await db
    .select()
    .from(orgVoicePersonas)
    .where(eq(orgVoicePersonas.organizationId, orgId))
    .limit(1)
    .then(rows => rows[0]);

  if (!persona) {
    throw new Error(`No voice persona configured for organization ${orgId}`);
  }

  // 2. If not opted in, return the static prompt unchanged.
  if (!persona.composeFromProfile) {
    return persona.systemPrompt;
  }

  // 3. Opted in. Load profile via the helper (uses withTenantTx + RLS).
  const profile = await getActivePracticeProfile(orgId);

  // 4. Fail-safe: if the org has no active profile, return the static prompt.
  //    Never send a prompt with the unsubstituted placeholder to Vapi.
  if (!profile || !profile.intake_field_specs) {
    return persona.systemPrompt;
  }

  // 5. Render the field list and substitute or append.
  const fieldsBlock = renderIntakeFieldsForPrompt(profile.intake_field_specs);

  if (persona.systemPrompt.includes('{{INTAKE_FIELDS}}')) {
    return persona.systemPrompt.replaceAll('{{INTAKE_FIELDS}}', fieldsBlock);
  }

  // No placeholder present — append at the end.
  return `${persona.systemPrompt}\n\n${fieldsBlock}`;
}
```

### 5b. `renderIntakeFieldsForPrompt`

Internal helper, not exported. Takes the `intake_field_specs` array (13 entries for Chajinel's profile) and produces a formatted block suitable for inclusion in a Vapi system prompt.

Format reference — match the tone of existing `system_prompt` content. Look at the current Chajinel persona's `system_prompt` to understand the existing style (clinical, conversational, instructional?) before designing the rendering. The rendering should feel like a continuation of the existing prompt, not a sudden style shift.

Suggested format (adjust to match observed style):

```
DATA TO CAPTURE DURING THE CALL:

Required fields:
- Caller first name
- Caller last name
- Phone
- Who needs care (the care recipient's name)
- Home address
- Hours of care needed per week
- How will care be paid for (insurance, VA, IHSS, private pay, LTC)

Optional fields (capture if naturally offered):
- Relationship to the care recipient
- ADL needs (bathing, dressing, mobility, etc.)
- Home environment notes
- Whether the caller is also considering assisted living
- VA authorization number
- IHSS county
```

Group by required vs optional. Render `display_label` (not `field_name`). Skip fields where `is_applicable === false` if that flag is set in the spec.

If an entry has a `help_text` field present, include it as a sub-bullet or italicized hint — match the surrounding prompt style.

### 5c. Edge cases

- Empty `intake_field_specs` array: return `system_prompt` unchanged with no appended block.
- Profile exists but has no `intake_field_specs` key (corrupt or partial): treat as empty, no append, no error.
- Persona row missing entirely for the org: throw — this is a configuration error caller must surface.

## Step 8 — Tests

Test cases:

1. **Opt-out path**: persona with `compose_from_profile=false` → builder returns `system_prompt` byte-identical.
