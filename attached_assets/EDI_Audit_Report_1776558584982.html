❌ Hard Errors — Will Cause Rejection
ERR-1 SV1-07 Diagnosis Pointer Must Be Numeric — 'A' Is Not Valid in EDI
This is the exact error Stedi flagged: "Element SV1-07-01 must use N0 number format; value found was 'A'". The X12 HIPAA 5010 spec requires SV1-07 (diagnosis pointer) to use numeric references — 1 2 3 4 — not the alphabetic letters used on the paper CMS-1500 form.

The claim wizard stores A B C D from the checkbox UI and that value flows directly into the EDI segment unchanged. This error will cause rejection at every clearinghouse that validates X12 element types.
FIX: Add a mapping function in edi-generator.ts before writing SV1: A→1, B→2, C→3, D→4. Apply it to all diagnosis pointer values before the SV1 segment is built.
ERR-2 HI ICD-10 Codes Contain Decimal Points — Not Allowed in X12
Two of the three diagnosis codes contain periods: G89.29 and Z51.11. X12 HIPAA 5010 requires ICD-10 codes in the HI segment to be transmitted without any decimal point. The correct form is G8929 and Z5111. Only I10 is correctly formatted.

This is a payer-level rejection — the claim will pass clearinghouse format validation but be rejected during payer adjudication because the code does not match any entry in the ICD-10 master file with the decimal still in.
FIX: In edi-generator.ts, strip all periods from ICD-10 codes before writing the HI segment: code.replace('.', '')
ERR-3 N4 — Line 18 Patient State 'CA' Contradicts City 'Austin' and ZIP 78749 (Texas)
The patient address shows N4*Austin*CA*78749. Austin is in Texas. ZIP code 78749 is a South Austin ZIP code. State CA (California) is inconsistent with both the city and ZIP. The billing provider address on line 12 correctly shows N4*Austin*TX*78749.

Root cause: the patient record was created using the state from the insurance card (a California-based BCBS plan) rather than the patient's physical address state. Payers run demographic validation — name, DOB, and address must match what the insurer has on file. A state mismatch causes eligibility validation failure.
FIX: Separate "insurance state" from "residential state" in the patient form. The N4 segment must use the patient's physical address state, not the insurance plan's home state. Update Daniela's patient record: State = TX.
ERR-4 SV1-01-2 HCPCS Code G0008 Is an Influenza Vaccine Admin Code — Not Home Health
The procedure code billed is G0008 — "Administration of influenza virus vaccine." This is a preventive care vaccine administration code with no clinical relationship to home health nursing visits. It was used during the demo to demonstrate code search functionality.

For Chajinel's skilled nursing home visits, the correct codes are:
  • G0299 — Direct skilled nursing services of a registered nurse (RN) in the home, per 15 minutes
  • G0162 — Skilled nursing for care plan management in the home, per 15 minutes
  • S9123 — Nursing care in the home, per hour (RN)
IMPACT: A G0008 claim submitted to VA Community Care will be denied immediately — it is not covered under a home health authorization. This has no coding basis whatsoever for a nursing visit.
ERR-5 NM1*41-09 Submitter ID Qualifier Is '46' (ETIN) But Value Is an NPI
The submitter loop uses qualifier 46 (Electronic Transmitter Identification Number — ETIN) but the value 1234567893 is the practice NPI. These are different identifiers. The ETIN is assigned by the clearinghouse during enrollment and is specific to the submitter's trading partner agreement. It is not the same as the NPI.

Office Ally assigns an ETIN during account setup. Stedi uses your API key or account ID as the submitter reference. Using the NPI in an ETIN field may cause routing failures at the interchange level before the claim is even parsed.
FIX: After Office Ally account setup, obtain the ETIN from Office Ally and store it in practice settings. Use that value in NM1-41-09. If the clearinghouse doesn't require a separate ETIN, use qualifier XX (NPI) instead of 46.
⚠ Warnings — Likely to Cause Denial or Processing Issues
WARN-1 NM1*DQ Ordering Provider NPI Equals Billing Provider NPI
The ordering provider (Loop 2310E) has NPI 1234567893 — identical to the billing practice NPI in NM1*85. The ordering provider on a home health VA claim must be the referring physician who ordered the home care — not the billing practice itself. VA Community Care requires a valid ordering physician NPI that is enrolled with them as a referring provider.
FIX: Daniela needs to provide the NPI of the VA physician who referred the patient for home health services. This should be a separate field in the patient record, not defaulting to the practice NPI.
WARN-2 NM1*82, NM1*DQ Provider Names Are Test Data Placeholders
Both provider names in this file are placeholder values — rendering provider Valid, Diana and ordering provider Valid, Alice. These are not real provider names and will fail NPI registry cross-validation at the payer. The NPI 1245319599 for Diana Valid would need to match a real enrolled provider.
FIX: Before the first real claim submission, Daniela must add her actual nurses with real NPIs in Settings → Providers. The demo provider data must not carry over into live claims.
WARN-3 CLM-09 Release of Information Code Is 'I' — Most Payers Expect 'Y'
CLM09 is set to I (informed consent to release medical information to payer). Most payers — including VA and Medicare — expect Y (provider has signed statement permitting release). The 'I' code is technically valid per X12 but some payers flag it during adjudication.
FIX: Change the hardcoded CLM09 value in edi-generator.ts from 'I' to 'Y'.
WARN-4 REF*EI Tax ID Is All Nines — Placeholder Value
The billing provider Tax ID in REF*EI is 123456789. This is a sequential placeholder, not a real EIN. The IRS EIN format is XX-XXXXXXX (9 digits). A sequential number like 123456789 will fail payer Tax ID validation and may trigger fraud screening flags.
FIX: Daniela's real EIN must be entered in Settings → Practice → Tax ID before any live claim is submitted.
