## Section 1: What this codebase is and what we are doing to it

ClaimShield is a React 18 + TypeScript + Express.js + PostgreSQL application being restructured into two distinct product modules within one codebase. The routing library is **Wouter** — not React Router, not Next.js. All route changes must use Wouter syntax.

**The two modules:**
- `ClaimShield Billing` — RCM, claim creation, denial management, analytics. Path prefix: `/billing/*`
- `ClaimShield Intake` — Lead pipeline, AI calling, SMS, email, chat widget, scheduling. Path prefix: `/intake/*`

Both modules share one PostgreSQL database, one Express backend, and one authentication system. They are separated by frontend route prefixes, role-based access control middleware, and distinct sidebar navigation components.

**What we are doing:**
1. Restructure existing routes into `/billing/*` and `/intake/*` namespaces using Wouter
2. Implement role-based access control that exists in the schema but is not enforced anywhere
3. Build the missing billing core loop on top of existing infrastructure
4. Fix broken integrations
5. Add missing database tables and columns without destroying existing data
6. Resolve all limiting factors identified in product review

**What we are NOT doing:**
- Rebuilding Vapi, Twilio, Gmail, or chat widget integrations — these work
- Rewriting the denial intelligence engine, risk scoring, or prior auth tracker — these work
- Dropping any tables or removing any columns
- Changing the auth system

### AuthGuard component

```tsx
function AuthGuard({ children, requiredRoles }: { 
  children: React.ReactNode, 
  requiredRoles?: string[] 
}) {
  const { user, isLoading } = useAuth();
  
  if (isLoading) return <LoadingScreen />;
  if (!user) return <Redirect to="/auth/login" />;
  
  if (requiredRoles && !requiredRoles.includes(user.role)) {
    return <Redirect to="/" />; // Back to module selector
  }
  
  return <>{children}</>;
}
```

## Section 4: Phase 0 — Structural fixes (complete before Phase 1)

### 0.1 Backend RBAC middleware

Add to the Express backend:

```typescript
function requireRole(...roles: string[]) {
  return (req: Request, res: Response, next: NextFunction) => {
    if (!req.user) return res.status(401).json({ error: 'Unauthenticated' });
    if (!roles.includes((req.user as any).role)) {
      return res.status(403).json({ error: 'Access denied' });
    }
    next();
  };
}
```

Apply to all route groups. Add `/api/billing/` prefix to all billing-related routes and `/api/intake/` prefix to all intake routes:

```typescript
// Billing routes
app.use('/api/billing/claims', requireRole('admin', 'rcm_manager'), claimsRouter);
app.use('/api/billing/patients', requireRole('admin', 'rcm_manager'), patientsRouter);
app.use('/api/billing/hcpcs', requireRole('admin', 'rcm_manager'), hcpcsRouter);
app.use('/api/billing/settings', requireRole('admin', 'rcm_manager'), settingsRouter);

// Intake routes
app.use('/api/intake/leads', requireRole('admin', 'intake'), leadsRouter);

```

### 0.2 VerifyTX timeout fix

The VerifyTX API is timing out. Apply these fixes to every VerifyTX API call:

```typescript
// Add timeout to all VerifyTX axios/fetch calls
const VERIFYTX_TIMEOUT_MS = 30000; // 30 seconds

// Retry once on timeout before returning error
async function verifytxRequest(url: string, options: RequestInit) {
  for (let attempt = 1; attempt <= 2; attempt++) {
    try {
      const controller = new AbortController();
      const timeout = setTimeout(() => controller.abort(), VERIFYTX_TIMEOUT_MS);
      const response = await fetch(url, { ...options, signal: controller.signal });
      clearTimeout(timeout);
      return response;
    } catch (err: any) {
      if (attempt === 2 || err.name !== 'AbortError') throw err;
      await new Promise(r => setTimeout(r, 5000)); // wait 5s before retry
    }
  }
}
```

Cache the payers directory response for 24 hours — store it in a `cache` table or as a simple JSON file. The payer list does not change hourly and should never need a live API call to render a dropdown.

Add a "Manual VOB" fallback to the VOB UI. When VerifyTX times out after both attempts, show:
> "Verification service unavailable — enter coverage details manually"

Manual VOB form fields: copay, deductible, out-of-pocket max, coverage effective date, coverage end date, network status (in-network / out-of-network / unknown), notes. Stored in the same `vob_verifications` table with a `source` field set to `'manual'` instead of `'verifytx'`.

### 0.3 VOB as shared infrastructure — dual-purpose design

VerifyTX serves both modules for different reasons. Do not duplicate the integration. Structure it as shared API routes under `/api/vob/`.

**Intake context** (lead qualification — does this patient's insurance cover home care?):
- Triggered from lead detail page
- Writes to `vob_verifications` table with `context: 'intake'`
- Updates `leads.vob_status` and `leads.vob_score`
- UI shows: covered services, copay, deductible, network status, go/no-go signal

**Billing context** (pre-claim eligibility — is coverage active for this service date?):
- Triggered from patient detail page (Eligibility tab) and from Step 3 of claim wizard
- Writes to `vob_verifications` table with `context: 'billing'`
- UI shows: active coverage dates, remaining deductible, authorization required flag, in-network status for the rendering provider's NPI

Both contexts query the same VerifyTX API and write to the same table. The `context` field distinguishes them. The billing eligibility tab on the patient detail view shows all VOB verifications for that patient, both intake and billing context, sorted by date.

When a lead is converted to a patient (see 0.4), the most recent intake-context VOB result is copied to the billing patient record automatically — the biller sees the intake verification without running a duplicate check.

### 0.4 Formal lead → patient conversion workflow

Replace the opportunistic patient creation with a deliberate conversion step.

**Current behavior** (keep, do not remove): Vapi call extraction creates/updates patient records automatically when insurance data is pulled from a call. This continues to work for pre-population.

**New behavior**: Add a "Convert to Patient" button on the lead detail page in the intake module. This button appears when `lead.status` is `'qualified'` or when `lead.vob_status` is `'complete'`.

On click, show a confirmation modal:

✓ Finalize the patient record with all collected information
✓ Send patient data to the billing module
✓ Mark this lead as converted

Patient data to transfer:
  Name:            [first_name] [last_name]

On confirm, the backend:
1. Upserts the patient record — if `patients` record exists for this `lead_id`, update it; if not, create it
2. Enriches the patient record with all data from the leads table (see database migration in Section 7)
3. Sets `leads.status = 'converted'` and `leads.handoff_status = 'completed'`
4. Logs to activity timeline: "Converted to patient — billing module notified"
5. Fires the intake → billing handoff (see 0.5)

**Providers tab** — table of providers with Add, Edit, Deactivate actions. NPI field validates with Luhn checksum:

```typescript
function validateNPI(npi: string): boolean {
  if (!/^\d{10}$/.test(npi)) return false;
  const withPrefix = '80840' + npi;
  let sum = 0;
  for (let i = 0; i < withPrefix.length; i++) {
    let digit = parseInt(withPrefix[withPrefix.length - 1 - i]);
    if (i % 2 === 1) { digit *= 2; if (digit > 9) digit -= 9; }
    sum += digit;
  }
  return sum % 10 === 0;
}
```

  COUNT(*) FILTER (WHERE status IN ('denied','suspended')) as denied_count,
