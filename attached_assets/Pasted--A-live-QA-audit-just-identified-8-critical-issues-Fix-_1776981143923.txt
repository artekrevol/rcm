### FIX 1A — Claim silently becomes "submitted" without 
Submit button being clicked

OBSERVED: Clicking "Download 837P EDI file" or 
"Download PDF" on a draft claim changes its status 
to "submitted" in the database. No Stedi transmission 
occurred. Clinics will see "submitted" and believe 
the claim went to the payer when it did not.

Find the PDF download and EDI download endpoints:
GET /api/billing/claims/:id/pdf-data
GET /api/billing/claims/:id/edi

Verify neither endpoint updates claim status.
Search for any UPDATE claims SET status = 'submitted'
or status = 'exported' in these endpoints.

The current code likely has:
UPDATE claims SET status = CASE 
  WHEN status IN ('draft','created') THEN 'exported' 
  ELSE status END

This is wrong. Downloading a file must NEVER change 
claim status to submitted or exported.

Fix: Remove all status updates from download endpoints.
Status must only change to 'submitted' when:
POST /api/billing/claims/:id/submit-stedi succeeds
OR
POST /api/billing/claims/:id/submit-oa succeeds

Add a status allowed-values check in submit endpoints:
If claim.status is already 'submitted': 
  return 400 "Claim already submitted"
If claim.status is 'paid':
  return 400 "Claim already paid — cannot resubmit"

After fixing: verify by downloading EDI and PDF for 
a draft claim and confirming status remains 'draft'.

To:
segments.push(`NM1*DK*1*...`)

the EXACT same format for test submission.

Console: "TypeError: a is not iterable at index-_1fI9JzZ.js"
Error originates in a useMemo hook.

Find reports.tsx. The error "a is not iterable" means 
a variable expected to be an array is null or undefined.

Common causes:

2. useMemo processes data before it loads
3. Destructuring fails on undefined response

Fix all four report queries to return empty arrays 
on no data, never null:

res.json({
  data: rows || [],        // never null
  summary: summary || {},  // never null
  filters: { ... }
});

// In reports.tsx, add null safety to all useMemo:
const chartData = useMemo(() => {
  if (!data?.data || !Array.isArray(data.data)) return [];
  return data.data.map(row => ({ ... }));
}, [data]);

// Check every .map(), .filter(), .reduce() call
// and add Array.isArray() guard before each

Find the specific useMemo that crashes by reading 
reports.tsx and identifying every array operation 
that lacks a null check.

After fixing: navigate to /billing/reports and verify 
all 4 tabs load without errors even with zero data.
