## Section 1: What this codebase is and what we are doing to it

ClaimShield is a React 18 + TypeScript + Express.js + PostgreSQL application being restructured into two distinct product modules within one codebase. The routing library is **Wouter** — not React Router, not Next.js. All route changes must use Wouter `Switch`, `Route`, and `Redirect` syntax exactly as the existing App.tsx uses them.

**The two modules:**
- `ClaimShield Billing` — RCM, claim creation, denial management, analytics. Path prefix: `/billing/*`
- `ClaimShield Intake` — Lead pipeline, AI calling, SMS, email, chat widget, scheduling. Path prefix: `/intake/*`

Both modules share one PostgreSQL database, one Express backend, and one Passport.js authentication system. They are separated by frontend route prefixes, role-based access control middleware, and distinct sidebar navigation components.

**What we are doing:**
1. Restructure existing Wouter routes into `/billing/*` and `/intake/*` namespaces
2. Implement role-based access control — roles exist in the schema but are enforced nowhere
3. Fix the logout flow — it currently only navigates to `/login` without destroying the session
4. Build the billing core loop on top of existing working infrastructure
5. Fix the VerifyTX timeout and add manual VOB fallback
6. Add missing database tables and columns without destroying existing data
7. Resolve all limiting factors identified in product review

**What we are NOT doing:**
- Rebuilding Vapi, Twilio, Gmail, or chat widget integrations — these work
- Rewriting the denial intelligence engine, risk scoring, or prior auth tracker — these work
- Dropping any tables or removing any columns
- Changing the Passport.js auth system structure

### Key observations — read before writing any code

**Patients table is very thin.** Nearly all demographic data lives on the `leads` table (first_name, last_name, phone, email, dob, state, insurance info). The billing module enriches the patients table via migration and the intake→billing handoff.

**No payers table exists.** Payer is stored as plain `text` on `claims.payer` and `patients.insurance_carrier`. The migration creates a normalized payers table. New claims use `payer_id` FK. Existing text fields are preserved.

**patients.dob is stored as text.** Do not cast it to a date type anywhere. Preserve as text throughout.

**claims.cpt_codes is JSONB.** The existing seeded data uses CPT codes (90834, 90837, 99213, 99214, 99215) stored as a JSON array. The new claim wizard uses HCPCS codes stored in a new `service_lines` JSONB column. Both coexist on the claims table — do not remove or overwrite cpt_codes.

**activity_logs has no claim_id or patient_id.** Only `lead_id`. The migration adds both nullable columns so billing events can be logged properly.

**vob_verifications already has both lead_id and patient_id.** The dual-module VOB design works without restructuring. Only a `context` column needs to be added.

**No auth guards exist.** Every route is currently accessible without authentication. The `AppSidebar` hardcodes "RCM Admin" and "admin@claimshield.health" with no role detection. Logout only navigates to `/login` without destroying the Passport.js session.

### The /api/leads/:id/claim-packet endpoint — deprecated demo stub

This endpoint creates claims with random CPT codes, random payer names ("Payor A/B/C/D/E"), and random dollar amounts. It is a prototype. **Do not extend or build on it.** The new claim wizard replaces it entirely.

Deprecate it by adding at the top of the handler:
```typescript
// DEPRECATED: This endpoint generates random demo data.
// Use POST /api/billing/claims/wizard instead.
console.warn('[DEPRECATED] /api/leads/:id/claim-packet called — use the claim wizard');
```

Do not delete it yet. Historical seeded records reference the encounter structure it created.

## Section 3: Pre-migration fixes — do these before running any migrations

These three fixes must be completed before anything else. They are quick and unblock everything that follows.

### Fix 1: Logout — destroy session properly

The current logout only navigates to `/login`. This must be fixed before adding auth guards or the guards can be bypassed.

**Backend — add a real logout route:**
```typescript
app.post('/api/auth/logout', (req, res) => {
  req.logout(() => {
    req.session.destroy(() => {
      res.clearCookie('connect.sid');
      res.json({ success: true });
    });
  });
});
```

**Frontend — update the logout handler in AppSidebar:**
```typescript
const handleLogout = async () => {
  await fetch('/api/auth/logout', { method: 'POST' });
  window.location.href = '/auth/login';
};
```

### Fix 2: Add context column to vob_verifications

The VOB table already has both `lead_id` and `patient_id` which is ideal. Add one column to distinguish intake-context from billing-context verifications:

```sql
ALTER TABLE vob_verifications
  ADD COLUMN IF NOT EXISTS context VARCHAR DEFAULT 'intake';
-- Values: 'intake' | 'billing'
-- All existing rows default to 'intake' which is correct
```

### Fix 3: Add claim_id and patient_id to activity_logs

The activity_logs table only has `lead_id`. Billing events (claim created, patient edited) need their own reference columns:

```sql
ALTER TABLE activity_logs
  ADD COLUMN IF NOT EXISTS claim_id VARCHAR,
  ADD COLUMN IF NOT EXISTS patient_id VARCHAR;
-- Both nullable — existing intake logs are unaffected
```

```tsx
// AuthGuard component
function AuthGuard({ 
  children, 
  requiredRoles 
}: { 
  children: React.ReactNode, 
  requiredRoles?: string[] 
}) {
  const { user, isLoading } = useAuth();

  if (isLoading) return <LoadingScreen />;
  if (!user) return <Redirect to="/auth/login" />;

  if (requiredRoles && !requiredRoles.includes(user.role)) {
    // Admin sees both — redirect to module selector
    // Single-role users are redirected to their module automatically
    return <Redirect to="/" />;
  }

  return <>{children}</>;
}
```

```tsx
// ModuleSelector — auto-redirect single-role users
function ModuleSelector() {
  const { user } = useAuth();

  // Auto-redirect if user only has access to one module
  if (user?.role === 'rcm_manager') return <Redirect to="/billing/dashboard" />;
  if (user?.role === 'intake') return <Redirect to="/intake/dashboard" />;

  // Admin sees both cards
  return (
    <div className="module-selector">
      <ModuleCard
        title="ClaimShield Billing"
        description="Build, submit, and track your claims"
        href="/billing/dashboard"
      />
      <ModuleCard
        title="ClaimShield Intake"
        description="Manage leads, calls, and patient intake"
        href="/intake/dashboard"
      />
    </div>
  );
}
```

```typescript
function requireRole(...roles: string[]) {
  return (req: Request, res: Response, next: NextFunction) => {
    if (!req.user) return res.status(401).json({ error: 'Unauthenticated' });
    if (!roles.includes((req.user as any).role)) {
      return res.status(403).json({ error: 'Access denied' });

}
```

### 0.2 VerifyTX timeout fix

Add timeout and retry to every VerifyTX API call:

```typescript
const VERIFYTX_TIMEOUT_MS = 30000;

async function verifytxRequest(url: string, options: RequestInit = {}) {
  for (let attempt = 1; attempt <= 2; attempt++) {
    try {
      const controller = new AbortController();
      const timeoutId = setTimeout(() => controller.abort(), VERIFYTX_TIMEOUT_MS);
      const response = await fetch(url, { ...options, signal: controller.signal });
      clearTimeout(timeoutId);
      return response;
    } catch (err: any) {
      if (attempt === 2 || err.name !== 'AbortError') throw err;
      await new Promise(r => setTimeout(r, 5000));
    }
  }
}
```

### 0.3 VOB dual-purpose design

The `vob_verifications` table already has both `lead_id` and `patient_id` — the dual-context design works without restructuring.

**Intake context** — triggered from lead detail, writes `context = 'intake'`, updates `leads.vob_status` and `leads.vob_score`.

**Billing context** — triggered from patient detail (Eligibility tab) and claim wizard Step 3, writes `context = 'billing'`.

**On patient conversion:** copy the most recent `vob_verifications` record for that lead (where `status = 'complete'` and `context = 'intake'`) to the billing patient record. Set `patients.vob_verified = true`. The biller sees the intake verification without re-running.

The Eligibility tab on patient detail shows all VOB records for that patient — both `context = 'intake'` and `context = 'billing'` — sorted by `verified_at` DESC, labeled separately.

```typescript
async function convertLeadToPatient(leadId: string) {
  const lead = await db.query('SELECT * FROM leads WHERE id = $1', [leadId]);
  
  // Upsert patient record — update if exists for this lead_id, create if not
  const patient = await db.query(`

      first_name, last_name, preferred_name, phone, email,
      referral_source, service_needed, intake_completed, vob_verified

      first_name = EXCLUDED.first_name,
      last_name = EXCLUDED.last_name,
      phone = EXCLUDED.phone,
      email = EXCLUDED.email,
      insurance_carrier = EXCLUDED.insurance_carrier,
      member_id = EXCLUDED.member_id,
      plan_type = EXCLUDED.plan_type,
      referral_source = EXCLUDED.referral_source,

    RETURNING *

    lead.first_name, lead.last_name, lead.preferred_name, lead.phone, lead.email,
    lead.source, lead.service_needed
  ]);
  
  // Update lead status
  await db.query(
    'UPDATE leads SET status = $1, handoff_status = $2 WHERE id = $3',

  });
  
  // Fire billing handoff
  await notifyBillingModule(patient);

}
```

```typescript
function validateNPI(npi: string): boolean {
  if (!/^\d{10}$/.test(npi)) return false;
  const withPrefix = '80840' + npi;
  let sum = 0;
  for (let i = 0; i < withPrefix.length; i++) {
    let digit = parseInt(withPrefix[withPrefix.length - 1 - i]);
    if (i % 2 === 1) { digit *= 2; if (digit > 9) digit -= 9; }
    sum += digit;
  }
  return sum % 10 === 0;
}
```

Query `activity_logs` for billing-related events (where `claim_id IS NOT NULL` or `patient_id IS NOT NULL`).

Columns: timestamp, performed_by, activity_type, description, linked record (link to claim or patient).

Filters: user, date range, activity_type.

**Compliance reports** (`/billing/intelligence/reports`):

Four report types using @react-pdf/renderer:

1. **Access report** — `SELECT * FROM activity_logs WHERE activity_type IN ('view_patient','view_claim') AND created_at BETWEEN $1 AND $2`
2. **Edit history report** — `SELECT * FROM activity_logs WHERE field IS NOT NULL AND created_at BETWEEN $1 AND $2`
3. **Export report** — `SELECT * FROM activity_logs WHERE activity_type = 'export_pdf' AND created_at BETWEEN $1 AND $2`
4. **Claims integrity report** — `SELECT c.*, p.first_name, p.last_name FROM claims c LEFT JOIN patients p ON c.patient_id = p.id WHERE c.created_at BETWEEN $1 AND $2`

Each: date range picker (default last 30 days), "Generate PDF" button, download link.
