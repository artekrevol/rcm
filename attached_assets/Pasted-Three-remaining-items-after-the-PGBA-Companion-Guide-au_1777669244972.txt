Before the demo I need a clear written assessment of what currently 
happens in ClaimShield AFTER a claim is submitted to Stedi. The 
pre-submission flow is well understood (intake → patient → claim 
wizard → risk score → 837P generation → Stedi submission). The 
post-submission flow has been built up across multiple sessions and 
I want a single source of truth on what's actually wired vs. stubbed 
vs. UI-only.

Trace the full post-submission lifecycle in the codebase and produce 
a written report covering:

a. SUBMISSION RESPONSE HANDLING
   - When test-stedi or submit-stedi returns from Stedi's API, what 
     happens to the response?
   - Is the Stedi transaction_id captured and stored on the claim 
     record? Which column?
   - Is the synchronous 999 acknowledgment from Stedi parsed and 
     surfaced anywhere in the UI?
   - What claim status transitions happen at this point?

b. 277CA ACKNOWLEDGMENT INGESTION
   - Which handler receives 277CAs — the polling job, the webhook, 
     or both? Which is primary?
   - Where does the 277CA parser live and what does it extract?
   - Is the rejectionCodeLookup service from the recent audit wired 
     into this path? If yes, where exactly (file + line). If no, what 
     would it take to wire it in.
   - When a 277CA arrives, what claim status transitions happen? What 
     events are written to the claim event timeline? What shows up in 
     the UI?
   - If the 277CA contains a rejection, is the user notified? How?

c. 835 ERA INGESTION AND POSTING
   - Which handler receives 835s — polling, webhook, or both?
   - Where does the 835 parser live?
   - Is payment auto-posting actually wired or is it stubbed? If wired,

     adjustment_amount, etc.)?
   - How are CARC codes captured and surfaced? Is the CARC lookup 
     library present and current?
   - Does the denial_patterns table get written to from real ERA data, 
     or is it still seeded data only?

     accepted, rejected, paid, denied, etc.) and the trigger for each

   - Are there any status values that exist in the schema but aren't 
     actually reachable in the live code path?
   - Identify any "stuck claim" detection logic — does it actually 
     fire alerts to the user, or is it UI-only display?

e. DENIAL INTELLIGENCE FEEDBACK LOOP
