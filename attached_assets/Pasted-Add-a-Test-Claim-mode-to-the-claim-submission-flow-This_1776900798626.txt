Add a "Test Claim" mode to the claim submission flow.

This uses Stedi's production API key with ISA15 set to 'T' 
to validate a claim end-to-end through Stedi's full 
validation pipeline without sending it to a real payer.

═══════════════════════════════════════════════════════
HOW STEDI TEST MODE WORKS
═══════════════════════════════════════════════════════
Stedi test API keys are only available for eligibility 
checks — they cannot be used for claim submission.

Instead, test claim submission works as follows:
- Use the same production API key (STEDI_API_KEY)
- Use the same production API endpoint
- Set ISA15 = 'T' (Test) instead of 'P' (Production) 
  in the 837P EDI string before sending
- Stedi's production environment accepts ISA15='T' 
  claims, runs full EDI validation, and returns a 
  real 277CA acknowledgment
- The claim is NEVER transmitted to the real payer
- This is the correct approach per X12 5010 spec — 
  ISA15 is the interchange usage indicator that tells 
  the receiving system whether this is test or 
  production data

To also test ERA (835) response:
- Set the payer ID in NM109 of Loop 2010BB to 'STEDI'
  (the Stedi Test Payer) instead of the real payer ID
- Stedi will return a test 835 ERA for this claim
- Use this specifically when testing ERA posting logic
- For general EDI validation, keep the real payer ID 
  and just use ISA15='T' — this validates routing 
  without generating a test ERA

Test submissions are free — Stedi only charges $0.20 
per claim for ISA15='P' (Production) submissions.

STEP 1 — New server function in stedi-claims.ts

Add a testClaim() function alongside the existing 
submitClaim() function:

export async function testClaim(
  params: StediClaimSubmissionParams
): Promise<StediSubmissionResult> {
  if (!STEDI_API_KEY) {
    throw new Error('STEDI_API_KEY not configured');
  }

  // Generate EDI same as production
  let ediContent = params.ediContent;

  // Set ISA15 to 'T' (Test) instead of 'P' (Production)
  // ISA15 is the 15th element of the ISA segment
  // ISA*00*          *00*          *ZZ*SENDER*ZZ*RECEIVER*DATE*TIME*^*00501*CTRL*0*P*:~
  //                                                                              ^ this is ISA15
  ediContent = ediContent.replace(
    /(\*[TP]\*)/,  // matches *P* or *T* at ISA15 position
    '*T*'
  );

  // Submit to Stedi production endpoint with ISA15='T'
  // This triggers test mode — no payer transmission
  const response = await fetch(STEDI_CLAIMS_URL, {
    method: 'POST',
    headers: {
      'Authorization': `Key ${STEDI_API_KEY}`,
      'Content-Type': 'application/json',
      'Idempotency-Key': `test-${params.claimId}-${Date.now()}`,
    },
    body: JSON.stringify({
      transactionSets: [ediContent],
    }),
  });

  const data = await response.json();

  if (!response.ok) {
    return {
      success: false,
      rawResponse: data,
      error: data.message || data.errors?.[0]?.description ||
             `Stedi test validation error: ${response.status}`,
    };
  }

  const txn = data.transactionSets?.[0] || data;
  const accepted = txn.status === 'Accepted' ||
                   data.status === 'Accepted' ||
                   response.status === 200;

  return {
    success: accepted,
    transactionId: txn.transactionId || data.transactionId,
    status: txn.status || (accepted ? 'Accepted' : 'Rejected'),
    validationErrors: txn.errors || data.errors || [],
    rawResponse: data,
    error: accepted ? undefined :
           (txn.errors?.[0]?.message || 'Claim failed Stedi validation'),
  };
}

STEP 2 — New backend endpoint

Add POST /api/billing/claims/:id/test-stedi
Role: admin, rcm_manager

Logic:
1. Fetch claim, verify org (same as submit-stedi)
2. Fetch patient, practice settings, provider, payer
3. Check isStediConfigured() — return 400 if not
4. Generate 837P EDI using existing generate837P()
5. Call testClaim({ ediContent, claimId }) 
   from stedi-claims.ts
6. Do NOT update claim status
7. Do NOT set follow_up_date
8. Do NOT create a 'Submitted' claim event
9. DO create a claim event:
   type = 'Test Validation'
   notes = `Stedi test validation: ${result.status}. 
            ${result.validationErrors?.length || 0} 
            validation issue(s) found.
            Transaction ID: ${result.transactionId}`
10. DO update a test_result column on the claim:
    ALTER TABLE claims 
      ADD COLUMN IF NOT EXISTS last_test_status VARCHAR,
      ADD COLUMN IF NOT EXISTS last_test_at TIMESTAMP,
      ADD COLUMN IF NOT EXISTS last_test_errors JSONB;
    
    UPDATE claims SET
      last_test_status = $status,
      last_test_at = NOW(),
      last_test_errors = $errors
    WHERE id = $claimId

11. Return:
{
  success: boolean,
  status: 'Accepted' | 'Rejected',
  transactionId: string,
  validationErrors: [
    {
      code: string,
      message: string,
      segment: string,    // which EDI segment failed
      field: string,      // which field within segment
    }
  ],
  summary: string        // human readable summary
}

═══════════════════════════════════════════════════════
IMPORTANT CONSTRAINTS
═══════════════════════════════════════════════════════

1. NEVER auto-submit after a passing test.
   The biller must explicitly click "Submit Claim" 
   as a completely separate action after seeing 
   the test result. The test modal should never 
   have a "Submit Now" button that is pre-selected 
   or defaulted — the biller must make a conscious 
   choice to submit.

2. Test submissions must NEVER change claim status 
   to 'submitted', 'acknowledged', or any status 
   that would appear in the Follow-Up Queue or 
   trigger the 277CA polling job.

3. The claim event created for a test submission 
   must be clearly labeled differently from real 
   submission events. Use type='Test Validation' 
   not type='Submitted'. The Claim Tracker timeline 
   should show test events in a different color 
   (grey, not blue) so billers can distinguish 
   test runs from real submissions.

4. Test submissions are free per Stedi's pricing 
   (ISA15='T' claims are not billed). Display this 
   clearly in the UI so billers use the feature 
   liberally rather than avoiding it to save money.

5. The Idempotency-Key for test submissions must 
   include a timestamp suffix (test-{claimId}-{Date.now()}) 
   so the same claim can be tested multiple times 
   without being deduplicated by Stedi.

6. If Stedi returns an unexpected response format 
   for test claims (their test pipeline may return 
   slightly different JSON than production), 
   log the raw response and show a fallback message:
   "Stedi returned an unexpected response. 
    Check the Activity Log for raw details."
   Never crash or show a blank modal.
