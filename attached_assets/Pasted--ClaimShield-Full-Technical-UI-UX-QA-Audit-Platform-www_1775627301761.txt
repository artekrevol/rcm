## Your role

You are a senior QA engineer and UX specialist conducting a full end-to-end audit of ClaimShield — a two-module medical billing SaaS for home health and home care agencies. The platform has two modules:

- **ClaimShield Intake** (`/intake/*`) — lead capture, qualification, AI outreach, insurance verification
- **ClaimShield Billing** (`/billing/*`) — claim creation, submission, denial management, compliance reporting

You will produce one structured audit report covering three areas:
1. **Technical QA** — every broken, inconsistent, or error-producing behavior
2. **UI/UX Audit** — every friction point, confusing pattern, or unnecessary step
3. **Flow Optimization** — the ideal simplified flow for each major workflow, preserving all compliance requirements

The report is the deliverable. No code changes until it is approved.

---

## How to conduct the audit

Work through each section methodically. For each area:

**A — Read the code** to understand the actual implementation, not assumptions
**B — Trace every data flow** from user action → API call → database → response → UI update
**C — Simulate every user journey** step by step as if you are operating the interface
**D — Check every failure path** — what happens when inputs are wrong, APIs fail, or data is missing
**E — Rate the UX** — would a non-technical home care office manager understand this on first use?

Use this scoring system throughout:
- 🔴 **Critical** — broken functionality, data loss risk, security issue, or compliance violation
- 🟡 **Major** — significantly hurts usability or causes frequent confusion
- 🟢 **Minor** — small improvement opportunity, low urgency
- ✅ **Pass** — working correctly, no issues

## 1.3 Database Health Check

Run these diagnostic queries and include the full results in the report:

```sql
-- 1. Orphaned claims (claim pointing to non-existent patient)
SELECT c.id, c.patient_id, c.status, c.amount
FROM claims c
LEFT JOIN patients p ON c.patient_id = p.id
WHERE p.id IS NULL;

-- 2. Claims missing critical fields
SELECT id, status, created_at
FROM claims
WHERE patient_id IS NULL
   OR amount IS NULL
   OR payer IS NULL
   OR status IS NULL;

-- 3. Patients with null names (broken conversion)
SELECT id, first_name, last_name, lead_id, intake_completed, created_at
FROM patients
WHERE first_name IS NULL AND last_name IS NULL;

-- 4. Users with unhashed passwords (security critical)
SELECT id, email, role, created_at
FROM users
WHERE password NOT LIKE '$2%';

-- 5. Activity logs with no record linkage
SELECT COUNT(*) as unlinked_logs
FROM activity_logs
WHERE claim_id IS NULL
  AND patient_id IS NULL
  AND lead_id IS NULL;

-- 6. Leads converted but with no corresponding patient
SELECT l.id, l.name, l.handoff_status, l.status
FROM leads l
WHERE l.handoff_status = 'sent'
AND NOT EXISTS (
  SELECT 1 FROM patients p WHERE p.lead_id = l.id
);

-- 7. VA location rates table health
SELECT COUNT(*) as total_rates,
  COUNT(DISTINCT location_name) as locations,
  COUNT(DISTINCT hcpcs_code) as codes
FROM va_location_rates;

-- 8. Session table exists and has active sessions
SELECT COUNT(*) as active_sessions,
  COUNT(*) FILTER (WHERE expire < NOW()) as expired_sessions
FROM session;

-- 9. HCPCS/ICD-10/CPT code counts
SELECT
  (SELECT COUNT(*) FROM hcpcs_codes) as hcpcs_codes,
  (SELECT COUNT(*) FROM icd10_codes) as icd10_codes,
  (SELECT COUNT(*) FROM cpt_codes) as cpt_codes;

-- 10. Duplicate payers
SELECT name, COUNT(*) as count
FROM payers
GROUP BY LOWER(name)
HAVING COUNT(*) > 1;
```

## Instructions for this section

For each major workflow below:

1. **Document the CURRENT flow** — every step, exactly as it works today
2. **Identify friction points** — steps that are redundant, confusing, or interruptive
3. **Propose the IDEAL flow** — minimum steps needed to accomplish the task while maintaining all compliance requirements
4. **Mark compliance-critical steps** — anything that must remain regardless of simplification (audit trails, required fields, authorization checks)
5. **Estimate impact** — what is the reduction in steps or time?

## 3.1 New Patient Intake → First Claim Flow

This is the full end-to-end user journey from a new referral to a submitted claim.

**Document and optimize:**
1. Lead enters system (chat / call / manual entry)
2. Lead is qualified (VOB completed)
3. Lead is converted to patient
4. Patient appears in billing
5. Claim is created for the patient
6. Claim is reviewed and exported/submitted

**Questions to address:**
- At what point should billing staff first become involved — step 3, 4, or 5?
- Can the draft claim pre-populate more fields from the lead data than it currently does?
- Is the handoff notification between intake and billing clear enough?
- How many screens does a biller touch between "patient appears in billing" and "claim exported"?
- Could steps 4 and 5 be collapsed into a single action?
