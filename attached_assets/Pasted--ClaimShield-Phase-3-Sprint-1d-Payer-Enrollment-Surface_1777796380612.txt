# ClaimShield — Phase 3 Sprint 1d: Payer Enrollment Surface Migration

This sprint migrates two production read paths from raw SQL to the helper service layer (`practice-profile-helpers.ts`), making them the first user-facing surfaces protected by RLS through `withTenantTx`. Discovery was completed in the planning conversation — the prompt below skips re-discovery and proceeds directly to implementation.

## Context (already established — do not re-investigate)

**The two surfaces being migrated:**

1. **Clinic settings page enrollment list** — `GET /api/practice/payer-enrollments` route handler at `server/routes.ts:3551–3571`. Returns the practice's payer enrollments to the frontend for display in clinic settings.
2. **Patient signup payer dropdown** — uses the same `GET /api/practice/payer-enrollments` endpoint (per Abeer's design choice a few days ago: "I asked Replit to use the same path to show the payers in the patient sign up flow"). Migrating the route handler migrates both surfaces simultaneously.

**Current behavior (verified against prod):** Both surfaces show Chajinel's 3 enrollments — Stedi Test Payer (FRCPB), TriWest Healthcare Alliance, VA Community Care. All three have `disabled_at = NULL`, all carry `organization_id = 'chajinel-org-001'`, all have `plan_product_code = NULL`.

**Source-of-truth alignment:** The current route handler's query and `getEnrolledPayers()` from `practice-profile-helpers.ts` resolve to the same row set for chajinel — no filter divergence. Migration is a refactor with one additive change to the helper.

**The one delta:** The current route returns two joined fields the helper doesn't currently expose:
- `payer_name` — already in helper (returned as `payerName`)
- `enrolled_by_name` — NOT in helper. Comes from `LEFT JOIN users u ON u.id = ppe.enrolled_by`. Needed to keep the settings page byte-identical.

## What this sprint is

1. Pre-flight: snapshot, baselines, confirm prod state matches discovery.
2. Extend `getEnrolledPayers()` to optionally include `enrolledByName: string | null` via a `LEFT JOIN users` in the helper query.
3. Migrate the `GET /api/practice/payer-enrollments` route handler to call the helper instead of raw `pool.query`.
4. Tests:
   - Helper test for `enrolledByName` resolution (3 cases: enrolled_by present + user exists, enrolled_by present + user deleted, enrolled_by null)
   - Regression test confirming the helper-backed route returns identical row sets to the pre-migration query (snapshot of expected rows for chajinel + demo, asserted against helper output)
5. Verification: existing baselines green, new tests green, tsc count ≤ 85.
6. Smoke: log into dev, open clinic settings page, confirm enrollments render. Open patient signup, confirm payer dropdown renders.
7. Audit report.
8. Migration-state update.

## What this sprint is NOT

- No changes to write-path handlers (`POST /api/practice/payer-enrollments` at `routes.ts:3573`, `DELETE` at `:3602`). Those stay on raw SQL until a separate sprint.
- No migration of `patient-detail.tsx`, `patient-create.tsx`, or other components that fetch from the same endpoint. They auto-migrate when the route handler flips because they consume the same response shape — but their broader UI test surface (form behavior, validation, etc.) is out of scope for sprint 1d.
- No changes to the helper's existing call sites (smoke-helpers.ts, sprint 0 verification). The helper extension is additive.
- No production deploy. Sprint 1d lands in dev. Production deploy is a separate task.
- No `replit_readonly` permission grants. (Still on the follow-up list.)
- No Drizzle organizations drift cleanup. (Still on the follow-up list.)
- No tsc baseline cleanup. (85-error baseline stays.)
- No Vapi work, no Caritas onboarding work, no PGBA work.

If a step seems to require touching a write handler, a non-settings-page component, or any unrelated code, stop and flag.

## Hard rules

1. Snapshot before any code change.
2. The migration touches exactly two files for behavior changes: `server/services/practice-profile-helpers.ts` (helper extension) and `server/routes.ts` (route handler migration). Plus one test file additions.
3. The route handler's response shape MUST match the pre-migration shape exactly. Same field names, same types, same null handling. Frontend code is not modified — it must continue to work without any change.
4. The helper extension is additive. Existing callers of `getEnrolledPayers()` (smoke-helpers, isolation tests) continue to work without modification.
5. The route handler must use `withTenantTx` so the query runs under RLS protection. The current raw-SQL route does not — that's part of what this sprint fixes.
6. Standing order: dev only. No production deploy in this sprint.
7. Stop on anomaly at any step.

## Step 3 — Migrate the route handler

### 3a. Read the current route handler

Read `server/routes.ts:3551–3600` (50 lines around the route). Document:
- The current handler's auth/middleware chain
- The exact SQL being run via `pool.query`
- The response shape (field names, types)
- Any error handling

### 3b. Replace the raw query with a helper call

The new handler calls `getEnrolledPayers()` from the helper. Because the helper goes through `withTenantTx`, the query runs under RLS — the org context comes from the tenant-context middleware that's already on every authenticated request post-Phase-3.

```typescript
// Pseudo-shape of the new handler:
app.get('/api/practice/payer-enrollments', requireAuth, async (req, res) => {
  try {
    const enrollments = await getEnrolledPayers();  // org context auto-resolved via withTenantTx
    res.json(enrollments);
  } catch (err) {
    // match existing error shape
  }
});
```

The exact shape depends on what's already there — match the existing handler's response envelope ({success: true, data: ...}, plain array, etc.).

### 3c. Verify response-shape parity

Before and after the migration, the response from the route should be byte-identical for the same input. The frontend doesn't change; the wire format must not change.

Run a quick comparison:

```bash
# Before changes (current commit):
curl -H "Authorization: Bearer <test-token>" http://localhost:5000/api/practice/payer-enrollments > /tmp/before.json

# Apply the migration

# After changes:
curl -H "Authorization: Bearer <test-token>" http://localhost:5000/api/practice/payer-enrollments > /tmp/after.json

diff <(jq -S . /tmp/before.json) <(jq -S . /tmp/after.json)
# Expected: zero diff
```

If they diverge, the migration changed wire behavior. Stop and reconcile.

### 3d. Add a regression test

In whichever file Replit chooses (probably `server/routes.test.ts` if it exists, or a new file), add a test asserting the route returns the expected row set for a known org context. This is the future-proofing for sprint 2+ — if anything in `getEnrolledPayers()` changes, this test catches the wire-format impact on the settings page.
