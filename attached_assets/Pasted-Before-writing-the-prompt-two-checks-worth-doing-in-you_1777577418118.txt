Message 1 — Architecture review request (send now or post-B2)

    Crawler Kit — Architecture Review Request (Not a Build Prompt)
    Context

    We've discussed payer manual ingestion as a competitive moat. Currently, the only paths into the corpus are (a) manual admin upload of PDFs and (b) Manus agent extraction (third-party SaaS). Long-term, we need ClaimShield to own this end-to-end with proprietary scrapers per payer, refreshing nightly, decommissioning Manus dependency.

    This is not a build greenlight yet. The build itself is post-May 12 work, ideally starting around May 14-15 once the Rosemary meeting and Chajinel advisor sign-off are done. What I need from you now is an architecture review — validate the approach below, identify gotchas, and produce a sequenced 3-week build plan I can greenlight after the meeting.
    Strategic frame

    ClaimShield's value proposition to investors and partners hinges on "the only billing platform with an automatic payer rules ingestion engine." This phrase only holds if the ingestion is actually automatic — manual upload doesn't sell it. The scraper kit is the operational backbone that makes the moat real.

    Specifically: every 90 days, the corpus drifts. Payers publish monthly bulletins, monthly medical policy updates, quarterly reimbursement policy updates, monthly prior auth list revisions. Without automated refresh, the corpus is stale by an average of 30-45 days within any 90-day window. With automated refresh, we can claim "Our corpus is fresher than any competitor's by an average of 23 days" — a defensible, measurable number that no other billing platform can claim.
    Proposed architecture
    Per-payer adapter pattern

    Not a generic crawler. One TypeScript file per payer in server/scrapers/, each implementing a common interface:
    typescript

    interface PayerScraper {
      payer_code: string;
      payer_name: string;
      
      // Returns the manifest of all documents available for this payer
      list_documents(opts?: { since?: Date }): Promise<DocumentManifest[]>;
      
      // Fetches a specific document by URL
      fetch_document(url: string): Promise<FetchResult>;
      
      // Optional: per-payer bulletin/news feed adapter
      list_bulletins(opts?: { since?: Date }): Promise<BulletinManifest[]>;
    }

    interface DocumentManifest {
      url: string;
      document_type: 'admin_guide' | 'supplement' | 'pa_list' | 'reimbursement_policy' | 'medical_policy' | 'bulletin';
      document_name: string;
      last_modified?: Date;
      content_hash?: string;
      discovered_at: Date;
      parent_document_url?: string;  // for supplements
    }

    interface FetchResult {
      content: Buffer;
      mimetype: string;
      final_url: string;  // after redirects
      fetched_at: Date;
    }

    Each adapter encodes the quirks of that payer's portal: URL patterns, HTML structure, login requirements, JavaScript rendering. They share nothing except the interface.
    Playwright, not basic fetch()

    Provider portals are hostile to basic HTTP clients. UHC has bot detection, JavaScript-rendered nav, session tokens. Aetna has authentication walls on most reimbursement policies. Cigna similar. Playwright handles all of this — at the cost of being slower and more expensive than raw HTTP. The tradeoff is necessary.
    Respectful crawling

        Rate limit: 1 request per 3-5 seconds per payer, configurable per adapter
        Honor robots.txt
        Identify User-Agent honestly: "ClaimShieldHealth/1.0 (compliance-research; contact@claimshield.health)"
        Exponential backoff on 429 / 503
        Circuit breaker: if a payer returns >5 consecutive errors, pause that payer's scraping for 24 hours and alert admin

    Storage and provenance

    Add columns to payer_source_documents:

        source_acquisition_method: enum of manual_upload, scraped, bulletin_triggered, manus_agent, cms_structured
        source_url_canonical: the URL the scraper fetched from
        content_hash: SHA-256 of the raw fetched bytes, used for change detection
        last_scraped_at: timestamp
        scrape_status: success, unchanged, error, auth_required, rate_limited

    Scheduled refresh

    Cron jobs (one per payer, staggered to avoid simultaneous load on any single payer's infrastructure):

        Nightly: list_documents() for the payer, diff against current payer_source_documents rows.
        For new documents: queue for ingestion via existing extraction pipeline.
        For changed documents (different content_hash): fetch, queue for re-extraction, flag existing extraction items as needs_reverification = TRUE.
        For unchanged: update last_scraped_at only.
        For errors: log, alert admin if persistent.

    Bulletin watchers

    Separate, simpler path. Most payers publish bulletins via RSS, Atom, or a structured "What's New" page. Per-payer bulletin adapter is dramatically simpler than full document scraping. Build these first — they generate the trigger signals that drive smart refresh of everything else.
    Manual upload remains first-class

    Some documents won't be scrapeable: login-walled portals requiring practice credentials, regional plan documents, contract-specific fee schedules. The admin UI already supports manual upload. Keep that path, tag with source_acquisition_method = 'manual_upload', and let practices upload anything the scrapers can't reach.
    Manus migration path

    Documents currently in the corpus that came from Manus agents get tagged source_acquisition_method = 'manus_agent'. As scrapers come online for each payer, we re-fetch via the scraper, compare hashes, and update provenance to scraped. Once a payer's scraper has successfully refreshed all its documents at least twice, mark the Manus dependency for that payer as deprecated. Goal: zero Manus dependency by end of Q3.
    Per-payer scope (for the 3-week build)

    Order matters. Start with hardest, finish with easiest:

    Week 1: UHC adapter. Most complex (484-page admin guide, 11+ supplements, separately-maintained PA list, hundreds of reimbursement policies, thousands of medical policies, monthly bulletins). If the pattern works for UHC, it works for everyone.

    Week 2: Aetna + Cigna adapters. Different login behaviors. Aetna especially has authentication walls that will test the credential-handling layer.

    Week 3: BCBS + Humana adapters + bulletin watchers across all five + scheduled refresh job. BCBS is structurally interesting because it's a federation — 36 independent BCBS plans with different portals. The adapter has to handle this as either one parent BCBS class with regional sub-adapters, or 36 separate adapters with shared utility functions. Pick a pattern and document it.
    What I need from you in this review

        Validate the per-payer adapter pattern. Is the interface right? Anything I'm missing in DocumentManifest or FetchResult?
        Playwright vs alternatives. Have you used Playwright in production for scraping at this scale? Any concerns about cost, reliability, or detection? Alternatives worth considering (Puppeteer, Selenium, headless Chrome with stealth plugins)?
        Login/credential handling. Some payer portals require provider credentials to access reimbursement policies. The practice has those credentials. How do we let practices supply credentials securely (encrypted at rest, never logged) and have the adapter use them per-fetch? Sketch the flow.
        Rate limiting architecture. With 5+ payers refreshing nightly, the rate-limit windows can stack up. Should this be in-process (per Node instance) or external (Redis-backed)? Replit's deployment context matters here.
        Failure mode handling. If UHC blocks our IP, what happens to the corpus? What's the fallback? How does the admin get alerted? Ideally there's a graceful degradation path that doesn't break ClaimShield for customers.
        Build sequence. I proposed Week 1 UHC, Week 2 Aetna+Cigna, Week 3 BCBS+Humana+bulletins+cron. Push back if you'd sequence differently. UHC first is intentional — if the pattern fails on UHC, we want to know in week 1, not week 3.
        Schema additions to payer_source_documents. I listed 5 new columns. Anything else needed for full provenance? effective_start and effective_end already exist from Prompt A.
        Per-payer document inventory. This is the part I can't generate alone. For each of UHC/Aetna/Cigna/BCBS/Humana, I need to provide you with the list of URLs and document types that matter. I'll work that up with Rosemary post-May 12. Confirm that's the right division of labor — you build the adapter framework, I supply the per-payer inventory.

    What I'm NOT asking for

        Code, yet. This is a planning review.
        A start date. Build kicks off post-May 12 (~May 14-15).
        Any change to the May 12 demo path. Cascade demo and B2 are sealed.

    Reply with the architecture review. I'll send the actual build prompt as Message 2 once we're past the Rosemary meeting and the Chajinel advisor signing.
