TASK: Make Member ID qualifier and value payer-driven in 837P generator

CONTEXT
The 837P generator currently emits the patient identifier as:
  NM1*IL*1*<last>*<first>****SY*<SSN>~

For VA CCN through TriWest/PGBA, this is wrong. It should be:
  NM1*IL*1*<last>*<first>*<middle>***MI*<EDIPI>~

The qualifier (SY vs MI) and the ID value (SSN vs EDIPI vs MBI vs commercial
member ID) varies BY PAYER, not by patient. Hardcoding either is wrong.

ARCHITECTURE
The qualifier belongs on the PAYER record. The value belongs on a
patient-payer association.

DATABASE CHANGES
1. Add to the payer record (or its config table):
     member_id_qualifier  VARCHAR(2) NOT NULL DEFAULT 'MI'
   Valid values per X12: 'MI' (Member ID), 'II' (Standard Unique Health ID),
   'SY' (SSN), 'ZZ' (Mutually defined). Most payers use 'MI'.

2. The patient identifier value should already be stored. If it's currently
   stored as a single "ssn" or "memberId" column on the patient record,
   that's fine — DO NOT migrate the data model in this ticket. Just rename
   the field semantically to "member_id" if it's currently called "ssn",
   and ensure it can hold non-SSN values like EDIPI (10 digits, not 9).

UI CHANGES
1. On the Payer record edit screen (under My Practice or Settings), add a
   field "Default Member ID Qualifier" with a dropdown of valid X12 values.
   Default: MI.
2. On the Patient record / insurance section, the field currently labeled
   "SSN" or similar should be re-labeled "Member ID" with helper text:
   "Use the ID expected by the payer — Medicare MBI, VA EDIPI, commercial
   subscriber ID, etc."

GENERATOR CHANGES (server/services/edi-generator.ts)
Find the NM1*IL emission for the subscriber/patient (Loop 2010BA).
Replace the hardcoded 'SY' with the qualifier from the payer config:

  Before:  `NM1*IL*1*${last}*${first}****SY*${ssn}~`
  After:   `NM1*IL*1*${last}*${first}*${middle || ''}***${payer.member_id_qualifier}*${memberId}~`

Note: middle name should be emitted in NM1*05 if present, otherwise the
position is empty (preserve the asterisk count).
