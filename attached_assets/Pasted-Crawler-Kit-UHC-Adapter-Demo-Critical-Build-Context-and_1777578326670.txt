Crawler Kit — UHC Adapter (Demo-Critical Build)
Context and stakes
The Rosemary meeting needs a live crawler demo running against real UHC content as the headline differentiator. The cascade demo proves the product UI. The crawler proves the operational engine behind it. Without the crawler, the meeting risks regressing because all engineering since the last meeting has been schema/infrastructure work invisible to a non-technical viewer.
This is a focused build scoped to UHC only. Other payers and document types are explicitly out of scope and come in subsequent prompts. The architecture must support N payers via the adapter pattern; we just only ship the UHC adapter in this prompt.
Critical constraint: start only after B2 is fully closed and verified. The crawler must not share files, services, or schema migrations with B2's in-flight work. If B2 has any open verification questions, those resolve first.
Architecture
Per-payer adapter pattern. All scrapers in server/scrapers/, each implementing the same interface. Common types in server/scrapers/types.ts. UHC-specific implementation in server/scrapers/uhc.ts.
typescript// server/scrapers/types.ts

export interface PayerScraper {
  payer_code: string;
  payer_name: string;
  
  list_documents(opts?: { since?: Date }): Promise<DocumentManifest[]>;
  fetch_document(url: string): Promise<FetchResult>;
  list_bulletins(opts?: { since?: Date }): Promise<BulletinManifest[]>;
}

export interface DocumentManifest {
  url: string;
  document_type: 'admin_guide' | 'supplement' | 'pa_list' | 'bulletin';
  document_name: string;
  last_modified?: Date;
  content_hash?: string;
  discovered_at: Date;
  parent_document_url?: string;  // links supplements to their admin_guide
}

export interface FetchResult {
  content: Buffer;
  mimetype: string;
  final_url: string;  // after redirects
  fetched_at: Date;
}

export interface BulletinManifest {
  url: string;
  title: string;
  published_at: Date;
  summary?: string;
  announces_changes_to?: string[];  // URLs of documents this bulletin updates, when discoverable
}

export interface ScrapeReport {
  payer_code: string;
  started_at: Date;
  completed_at: Date;
  documents_discovered: number;
  documents_new: number;
  documents_updated: number;
  documents_unchanged: number;
  bulletins_discovered: number;
  errors: { url: string; error: string }[];
  used_fallback: boolean;  // true if live scrape failed and cached result was served
}

T2 — Scraper runtime infrastructure
Install Playwright (npm install playwright) and run npx playwright install chromium.
Create server/scrapers/runtime.ts exposing:
typescriptexport async function withBrowser<T>(
  fn: (page: Page) => Promise<T>,
  opts?: { timeoutMs?: number; userAgent?: string }
): Promise<T>;

export async function rateLimit(payer_code: string): Promise<void>;
export function resetRateLimit(payer_code: string): void;

export async function checkCircuit(payer_code: string): Promise<'closed' | 'open' | 'half_open'>;
export async function recordSuccess(payer_code: string): Promise<void>;

Behavior:

Browser config: Chromium headless. User-Agent: ClaimShieldHealth/1.0 (compliance-research; contact@claimshield.health). Default navigation timeout 30 seconds. Disable resource loading for images/fonts/media to speed up parsing (only HTML and JSON needed).
Rate limiter: in-memory token bucket per payer_code. Default: 1 request per 4 seconds. Configurable per adapter via constructor option.
Circuit breaker: persisted in scraper_circuit_state. After 5 consecutive errors within a 10-minute window, transition state to open, set reopens_at = now() + 24 hours. While open, all calls return early with circuit_open. Half-open state on first attempt after reopens_at. Successful call closes the circuit.
Robots.txt pre-flight: before any session, fetch ${payer_domain}/robots.txt, cache for 24 hours per payer. Log if intended URLs are disallowed but do not block (document this decision in code comments — we're identifying ourselves honestly with our User-Agent and respecting rate limits, so blocking ourselves on robots.txt heuristics is over-cautious for compliance research crawling).

Critical implementation notes:

UHC's site is JavaScript-rendered for navigation. Playwright handles this; basic fetch will return empty pages.
The admin guides page changes structure year-over-year. Make the parser resilient — match on link text patterns and href containing .pdf, not on hardcoded DOM positions.
PDF URLs on UHC's CDN sometimes redirect through download handlers. fetch_document follows redirects (Playwright default) and captures the final_url.
PDFs exceeding Claude's 100-page vision limit are out of scope for fixing here — the existing extraction pipeline already handles oversized PDFs with a clear error message. The crawler just fetches; the existing pipeline's size handling is unchanged.
Hash strategy: SHA-256 of the raw fetched bytes. Compute during fetch_document and return as part of FetchResult (extend the type to include content_hash).

Add an in-process fallback cache at server/scrapers/uhc-fallback-cache.ts:

Stores the result of the most recent successful list_documents() and a sample fetch_document() for the Capitation/Delegation Supplement.
Updated automatically on every successful run.
Used as a fallback when live scraping fails during the demo path (T4 handles the fallback orchestration).
This is critical: if UHC's site is slow or returns errors during the demo, the fallback ensures the demo still completes successfully. The demo button surfaces a small badge ("Cached") when fallback is used, so the user sees what happened — but the demo still works.

T4 — Discovery + ingestion job
Create server/jobs/scrape-payer-documents.ts:
typescriptexport async function scrapePayerDocuments(
  payer_code: string,
  opts?: {
    since?: Date;
    dryRun?: boolean;
    allowFallback?: boolean;     // default true; demo button always uses true
    triggeredBy?: 'manual_admin' | 'cron' | 'demo_button';
    userId?: string;
  }
): Promise<ScrapeReport>;
Algorithm:

Insert row into scrape_runs with status = 'running'.
Check circuit breaker. If open: return early, status circuit_open, mark scrape_runs.status = 'circuit_open'.
Resolve scraper for payer_code (only 'uhc' registered for now; throw with clear error if unknown).
Try scraper.list_documents({ since }). On success, proceed to step 5. On failure: if allowFallback, load the fallback cache and proceed with cached manifest, set used_fallback = true. If !allowFallback, record error and abort.
For each manifest entry, look up by source_url_canonical in payer_source_documents:

Not in DB: call scraper.fetch_document(url). Compute SHA-256. INSERT new payer_source_documents row with:

source_acquisition_method = 'scraped'
source_url_canonical = url
content_hash = sha256
scrape_status = 'success'
last_scraped_at = now()
status = 'pending' (existing column — triggers extraction)
payer_id = (SELECT id FROM payers WHERE LOWER(name) LIKE 'united%' LIMIT 1) — for UHC; resolved per adapter


For supplements: also resolve parent_document_id by matching the manifest's parent_document_url against existing source_url_canonical. NULL if not found yet (will be backfilled on a subsequent run).
Trigger extraction by calling the existing manual-extractor function directly (not via HTTP).
In DB, hash matches: UPDATE last_scraped_at = now(), scrape_status = 'unchanged'. No re-extraction.
In DB, hash differs (new version published): Mark prior row as superseded by setting status = 'superseded', effective_end = today - 1. INSERT new row as above with parent_document_id = prior.id (so version history is traceable). Trigger extraction. UPDATE all extraction items belonging to the prior version: needs_reverification = TRUE, append note: "Superseded by document version scraped at [timestamp]. Re-review required.".


Try scraper.list_bulletins({ since }). For each new bulletin URL not already in DB:

INSERT payer_source_documents with document_type = 'bulletin', source_acquisition_method = 'bulletin_triggered'. Do not auto-trigger extraction (bulletin extraction logic is out of scope; rows queue for admin review).


UPDATE scrape_runs row: completed_at, status (success / partial / failed depending on errors), report (full JSONB), used_fallback.
On any error path: recordError(payer_code, err) to drive circuit breaker.
Return ScrapeReport.

Concurrency: a payer can have only one running scrape at a time. If a second call comes in while one is running, return early with status: 'already_running' in the report.
T5 — API endpoints
All require requireSuperAdmin except where noted:

POST /api/admin/scrapers/run — body { payer_code, dryRun?, allowFallback? }. Triggers scrape. Returns { run_id, status: 'running' }. Response is fast; report polled separately.

GET /api/admin/scrapers/status — returns per-payer status: circuit state, last successful run timestamp, documents tracked count, last 5 run summaries.
POST /api/admin/scrapers/circuit/:payer_code/reset — manual circuit reset. Logs the reset reason.
GET /api/admin/scrapers/discoveries — paginated feed of newly-discovered documents in the last N days (default 30), filterable by payer + document_type.

What this prompt does NOT do

Other payers (Aetna, Cigna, BCBS, Humana). Same adapter pattern, separate prompts.
Reimbursement policies and medical policies. Different ingestion pipeline. Separate prompt.
Login/credential handling. UHC's public docs don't require it; other payers will. Separate prompt when first credentialed payer is added.
Scheduled cron refresh. Manual trigger and demo button only. Cron job comes after the meeting and proves out the daily refresh story.
Production-grade rate limiting (Redis-backed). In-memory is correct for single-instance deploy; Redis migration is a follow-up when multi-instance is needed.
Bulletin extraction pipeline. Bulletins are discovered and stored; extraction logic for them is a separate prompt.
Backfilling parent_document_id for orphaned supplements after first scrape. The first scrape may discover supplements before their admin_guide. A second scrape will have both rows present and can resolve. Acceptable.

Failure-mode requirements (do not skip)

Live scrape failure must not break the demo. The fallback cache exists for one reason: the demo button always succeeds. Test this explicitly in T7.15.
Circuit breaker triggering must not require code intervention. A blocked IP or a UHC site change should pause scraping for 24 hours, alert the admin via the status endpoint, and not require redeployment to recover.
No silent failures. Every error path must either be reported in the ScrapeReport or surfaced in the SSE stream. If a fetch fails silently, the admin has no signal anything's wrong, and the corpus drifts.
Idempotency. Running the same scrape twice must produce no duplicate rows, no duplicate extraction triggers, no duplicate runs in flight.
