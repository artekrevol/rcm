# ClaimShield — Phase 3 Production Deploy

This prompt deploys the combined output of Phase 3 Sprint 0, Sprint 1a, and Sprint 1b to production. Everything has been validated in dev (`heliumdb`); production (`PRODUCTION_DATABASE_URL`, Railway-managed) has none of it yet.

**This is the riskiest single change to ClaimShield to date.** It introduces:
- 6 new database tables (5 tenant-scoped, 1 global catalog)
- 12 RLS policies + FORCE RLS
- 2 new database roles
- 11 additive ALTER COLUMN statements on a live table (`practice_payer_enrollments` has 3 production rows)
- 1 additive ALTER COLUMN on another live table (`org_voice_personas`)
- 2 seed rows
- A new Express middleware on the production boot path that wraps every authenticated request in a transaction
- Application code from sprints 0, 1a, 1b: helpers, Tier 1 validator wired into `evaluateClaim`, voice persona builder, Drizzle schema corrections

After deploy, every authenticated request in production runs through `tenantContextMiddleware` → `withTenantTx` → `SET LOCAL ROLE claimshield_app_role`. The architecture stops being conceptual and becomes the production runtime.

## Hard rules

1. **Schema migration runs before code deploy, not after.** Prod's app code currently uses old Drizzle definitions; sprint 0/1a/1b code expects the new schema. If code deploys first, the app fails to boot. Strict ordering: snapshot → schema → verify → code → verify.
2. **Every irreversible step has an Abeer sign-off gate.** Eight gates total. Replit pauses, surfaces the diff, waits for explicit approval before continuing. No batched approvals.
3. **All commits before code deploy go to a `phase3-prod-deploy` branch, not `main`.** Railway only deploys from `main`. Staging on a separate branch keeps code out of production until the gates pass.
4. **Snapshot before any DDL.** `pg_dump` of production saved to `/tmp/` (NOT committed to repo, NOT in `docs/architecture/`). Path recorded in audit report. This is the rollback artifact.
5. **All schema changes are additive.** No drops, no renames, no type changes. Existing queries continue to work during and after migration.
6. **Tenant-scoped data is preserved exactly.** The 3 production rows in `practice_payer_enrollments` (Chajinel's TriWest, VA Community Care, Stedi Test Payer enrollments) are not modified. Any modification is a critical bug.
7. **No PHI or production data is logged, screen-printed, or written to any file outside the snapshot.** Patient names, member IDs, claim contents — none of it appears in audit reports, console logs, or migration scripts.
8. **If anything looks wrong at any gate, stop and surface.** Do not "fix and continue." This is production.

## What this deploy is

1. Pre-flight: production state inspection, baseline metrics, snapshot, connection role identity check.
2. Stage code on `phase3-prod-deploy` branch (not deployed yet).
3. Apply schema migration to production via controlled DDL script.
4. Verify schema migration.
5. Merge code to `main`, trigger Railway deploy.
6. Verify code deploy.
7. Smoke test against production.
8. Audit report.

## What this deploy is NOT

- No new feature work. Code being deployed is exactly what's already merged in dev across sprints 0, 1a, 1b — no additions, no fixes, no improvements during the deploy.
- No EDI route gate (sprint 1c work).
- No payer dropdown migration to use `getEnrolledPayers()` (sprint 1c work — for now, the production dropdown continues using whatever query Abeer configured a few days ago).
- No Caritas onboarding (post-deploy task).
- No Vapi assistant activation.
- No `replit_readonly` permission grants on `practice_payer_enrollments` or `org_voice_personas` (separate small task; can be bundled at the end if Abeer chooses).
- No connection pool size changes (monitor post-deploy; adjust only if exhaustion observed).

If any of the above look attractive during deploy because the prod connection is open, **stop**. Add to a follow-up task list. Do not expand scope mid-deploy.

## Phase 1 — Pre-flight (read-only against production)

### 1a. Connection role identity — the most important pre-flight check

The sprint 0 RLS + role pattern assumes the application's database connection is a superuser (in dev, it's `postgres`). Production may differ — Railway-managed Postgres sometimes provisions a non-superuser application connection.

```sql
-- Run against PRODUCTION_DATABASE_URL
SELECT current_user, session_user, current_database(), version();
SELECT rolname, rolsuper, rolcreaterole, rolcreatedb, rolbypassrls
FROM pg_roles WHERE rolname = current_user;
```

Three possible outcomes:

**Outcome A — current_user is superuser (`rolsuper = true`):** Same pattern as dev. `claimshield_app_role` is created, granted to the connecting role, `SET LOCAL ROLE` works as designed.

**Outcome B — current_user is not superuser but has CREATE ROLE privilege:** The migration can still create `claimshield_app_role`, but the grant pattern needs adjustment. Document the exact role and its privileges, propose a revised role design, **stop and surface to Abeer**.

**Outcome C — current_user is not superuser and lacks CREATE ROLE:** The schema migration cannot run as-is. Need a separate Railway-admin connection (or Railway support intervention) to create the role. **Stop and surface to Abeer.** Do not proceed with the deploy until role creation path is determined.

Whatever the outcome, document in `docs/architecture/phase3-deploy-preflight.md` §1.

**Sign-off gate 1: connection role identity confirmed.** Abeer reviews outcome, approves the role-creation path. Continue to 1b only after approval.

### 2c. Build the deploy migration script

Create `scripts/phase3-deploy-prod.sql` on the branch. This is the single source of truth for the schema migration. It contains every DDL statement needed, in order, idempotent where possible (`IF NOT EXISTS`, `IF EXISTS`).

The script structure:

```sql
-- scripts/phase3-deploy-prod.sql
-- Phase 3 production migration. Run via:
--   psql $PRODUCTION_DATABASE_URL -f scripts/phase3-deploy-prod.sql
-- Or:
--   railway run psql -f scripts/phase3-deploy-prod.sql
--
-- This script is idempotent up to the seed rows (which use ON CONFLICT DO NOTHING).
-- All schema changes are additive — no drops, no renames, no type changes.

BEGIN;

-- ========== 1. Roles ==========
-- Adjust the GRANT TO statement based on Phase 1 connection role outcome.

DO $$
BEGIN
  IF NOT EXISTS (SELECT 1 FROM pg_roles WHERE rolname = 'claimshield_service_role') THEN
    CREATE ROLE claimshield_service_role NOINHERIT;
  END IF;
  IF NOT EXISTS (SELECT 1 FROM pg_roles WHERE rolname = 'claimshield_app_role') THEN
    CREATE ROLE claimshield_app_role NOLOGIN NOINHERIT;
  END IF;
END $$;

-- Grant the app role to the connecting user so SET LOCAL ROLE works
-- The role name in the GRANT below must match Phase 1 outcome:
--   Outcome A (superuser): GRANT claimshield_app_role TO postgres;
--   Outcome B/C: GRANT claimshield_app_role TO <whatever_role>;
-- This line is filled in based on the preflight check.
GRANT claimshield_app_role TO <CONNECTING_ROLE>;  -- placeholder

-- ========== 2. New tables (CREATE IF NOT EXISTS for idempotency) ==========
-- (paste exact DDL from sprint 0 audit's docs/architecture/sprint0-snapshots/sprint0-ddl.sql,
--  unchanged from dev. The DDL is already audited and verified.)

-- practice_profiles
CREATE TABLE IF NOT EXISTS practice_profiles (
  -- ... per shared/schema.ts and sprint0-ddl.sql
);

-- organization_practice_profiles
CREATE TABLE IF NOT EXISTS organization_practice_profiles (
  -- ...
);

-- (and so on for the other 4 relationship tables)

-- ========== 3. Additive ALTERs to practice_payer_enrollments ==========
-- 11 additive columns, all NOT NULL with safe defaults or NULL-able.

ALTER TABLE practice_payer_enrollments
  ADD COLUMN IF NOT EXISTS enrollment_status text NOT NULL DEFAULT 'pending',
  ADD COLUMN IF NOT EXISTS effective_from date,
  ADD COLUMN IF NOT EXISTS effective_to date,
  ADD COLUMN IF NOT EXISTS billing_npi text,
  ADD COLUMN IF NOT EXISTS taxonomy_code text,
  ADD COLUMN IF NOT EXISTS submission_method text,
  ADD COLUMN IF NOT EXISTS clearinghouse text,
  ADD COLUMN IF NOT EXISTS timely_filing_days integer,
  ADD COLUMN IF NOT EXISTS prior_auth_required boolean NOT NULL DEFAULT false,
  ADD COLUMN IF NOT EXISTS contracted_rate_table_id uuid,
  ADD COLUMN IF NOT EXISTS created_at timestamptz NOT NULL DEFAULT NOW(),
  ADD COLUMN IF NOT EXISTS updated_at timestamptz NOT NULL DEFAULT NOW();

-- ========== 4. Additive ALTER on org_voice_personas ==========
ALTER TABLE org_voice_personas
  ADD COLUMN IF NOT EXISTS compose_from_profile boolean NOT NULL DEFAULT false;

-- ========== 5. Enable RLS + FORCE RLS on the 6 tenant-scoped tables ==========
ALTER TABLE organization_practice_profiles ENABLE ROW LEVEL SECURITY;
ALTER TABLE organization_practice_profiles FORCE ROW LEVEL SECURITY;
-- (repeat for the 5 others)

-- ========== 6. Policies ==========
-- 12 policies: tenant_isolation + service_role_bypass on each of the 6 tenant tables.
-- Use the EXACT GUC name verified from dev (likely `app.current_organization_id` —
-- confirm via dev pg_policy before pasting here).
-- Both USING and WITH CHECK on tenant_isolation (sprint 1a closed this).

CREATE POLICY tenant_isolation ON organization_practice_profiles
  USING (organization_id = current_setting('app.current_organization_id', true))
  WITH CHECK (organization_id = current_setting('app.current_organization_id', true));
-- (repeat for the 5 other tenant tables)

CREATE POLICY service_role_bypass ON organization_practice_profiles
  AS PERMISSIVE FOR ALL TO claimshield_service_role
  USING (true) WITH CHECK (true);
-- (repeat for the 5 others)

-- ========== 7. Grants ==========
-- claimshield_app_role
GRANT SELECT, INSERT, UPDATE, DELETE ON
  organization_practice_profiles, practice_payer_enrollments,
  provider_practice_relationships, provider_payer_relationships,
  patient_insurance_enrollments, claim_provider_assignments
TO claimshield_app_role;

GRANT SELECT ON practice_profiles TO claimshield_app_role;

-- Parent-table SELECT grants (sprint 0 mid-sprint addition; required for helper joins)
GRANT SELECT ON organizations, payers, providers, patients, claims TO claimshield_app_role;

-- claimshield_service_role
GRANT SELECT, INSERT, UPDATE, DELETE ON
  organization_practice_profiles, practice_payer_enrollments,
  provider_practice_relationships, provider_payer_relationships,
  patient_insurance_enrollments, claim_provider_assignments
TO claimshield_service_role;

-- ========== 8. Seed: practice_profiles + Chajinel mapping ==========
-- (paste the exact INSERT for home_care_agency_personal_care from sprint 0 §2.3,
--  unchanged from dev. Use ON CONFLICT (profile_code) DO NOTHING.)

INSERT INTO practice_profiles (profile_code, ...) VALUES (
  'home_care_agency_personal_care', ...
) ON CONFLICT (profile_code) DO NOTHING;

INSERT INTO organization_practice_profiles (organization_id, profile_code, is_primary)
VALUES ('chajinel-org-001', 'home_care_agency_personal_care', true)
ON CONFLICT DO NOTHING;

COMMIT;
```

## Phase 6 — Rollback (documented, not executed unless needed)

### 6a. Code rollback (fast, < 5 minutes)

```bash
git checkout main
git revert -m 1 <merge-commit-sha>  # the Phase 4 merge commit
git push origin main
```

Railway picks up the revert and redeploys the prior version. The middleware is removed from the production boot path. The app reverts to pre-Phase-3 behavior.

### 6b. Schema rollback (slower, manual)

If the schema also needs to revert (because the new columns or tables are causing issues even with the old code):

The new tables can be dropped:
```sql
DROP TABLE IF EXISTS claim_provider_assignments;
DROP TABLE IF EXISTS patient_insurance_enrollments;
DROP TABLE IF EXISTS provider_payer_relationships;
DROP TABLE IF EXISTS provider_practice_relationships;
DROP TABLE IF EXISTS organization_practice_profiles;
DROP TABLE IF EXISTS practice_profiles;
```

The new columns on `practice_payer_enrollments` and `org_voice_personas` can stay (additive, not breaking). Or drop them if needed:
```sql
ALTER TABLE practice_payer_enrollments
  DROP COLUMN enrollment_status,
  DROP COLUMN effective_from,
  -- (etc.)
;
```

The roles can stay or be dropped:
```sql
DROP ROLE IF EXISTS claimshield_app_role;
DROP ROLE IF EXISTS claimshield_service_role;
```

If any rollback step fails, the snapshot at `/tmp/claimshield-prod-pre-phase3-*.sql` is the last resort — full database restore. This is a major operation with downtime; only use if all other rollback paths fail.
